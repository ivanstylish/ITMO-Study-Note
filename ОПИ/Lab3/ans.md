# ЧАСТЬ 1: СИСТЕМЫ АВТОМАТИЧЕСКОЙ СБОРКИ

## Зачем нужна автоматическая сборка?

При разработке Java-проекта нужно постоянно повторять одни и те же команды:
```
javac -d build -cp lib src/*.java    // компиляция
cp src/*.properties build            // копирование ресурсов
jar cfm app.jar manifest.cf build    // упаковка в архив
```

Делать это вручную — долго, скучно и можно ошибиться. Поэтому придумали **системы автоматической сборки**.

**Что умеют системы сборки:**
- Используют специальные языки для описания компиляции
- Задают конфигурацию системы и автоматически определяют архитектуру
- Могут делать сборку декларативно (говоришь ЧТО хочешь, а не КАК делать)
- Поддерживают параллельный и многомашинный режим
- Интегрируются с CI/CD серверами (GitLab CI, Jenkins) и автотестами

---

## Проблемы сборки (три "ужаса")

### Ужас №1 — Рутинный процесс
Каждый раз вручную повторять одни и те же команды. Скрипты ОС помогают, но их сложно поддерживать — любое изменение в структуре проекта нужно вручную отражать в скрипте.

### Ужас №2 — Отличия архитектур систем
Среда разработки почти всегда отличается от среды выполнения:
- Разная разрядность процессора (32/64 бит)
- Разные библиотеки и их расположение в ОС
- Разные стандарты (POSIX и др.)
- Пример: файл `/etc/hosts` в Ubuntu лежит по одному пути, в Windows — по другому

### Ужас №3 — Медленная сборка
- Много файлов → долгая компиляция (особенно C++)
- Придумали **ночные сборки** — запускать сборку ночью когда никто не работает
- Не нужно каждый раз компилировать всё — только изменённые файлы (инкрементальная сборка)
- Современные системы поддерживают **параллельную сборку** на нескольких процессорах

## Make и Makefile

**Make** — первый инструмент автоматизации сборки, появился в Unix.

В `Makefile` описывают:
- **Цели** (targets) — что нужно собрать
- **Зависимости** — от каких файлов зависит цель
- **Команды** — что выполнить для сборки

```makefile
OBJS = tlm
all: ${OBJS} tlm64 tlmctl

tlm.o: tlm.h tlm_subs.c tlm.c
    ${CC} ${CFLAGS} -c -o tlm.o

clean:
    rm -f core ${OBJS} *.o
```

**Как работает Make:**
1. Читает Makefile начиная с цели `all`
2. Проверяет время последней модификации файлов
3. Если файл изменился — пересобирает только зависимые от него части
4. Это экономит время — не нужно компилировать всё с нуля

**Минус Make:** сложный синтаксис, много макропеременных, платформозависимый.

## Apache Ant

**Ant** — система сборки для Java. Управляется файлом `build.xml` в формате XML.

**Принцип:** похож на Make, но **императивный** (описываешь КАК делать шаг за шагом).

```xml
<project name="World" basedir=".">
    <target name="init">...</target>
    <target name="build" depends="init">...</target>
    <target name="dist" depends="build">
        <antcall target="..."/>
    </target>
</project>
```

**Ключевые понятия Ant:**

### Свойства (Properties) — переменные
Задаются тремя способами:
```xml
<!-- Значением напрямую -->
<property name="build.dir" value="/tmp"/>

<!-- Переменной окружения -->
<property environment="env"/>
<echo message="Path: ${env.PATH}"/>

<!-- Из файла (самый удобный способ) -->
<property file="build.properties"/>
```
Использование: `${build.dir}` — подставляет значение переменной.

**Важно:** свойства в Ant **неизменяемые** — раз установил, уже не поменяешь.

### Команды (Tasks) Ant — что умеет делать:
- **Файловые операции:** copy, delete, move, mkdir, checksum
- **Архивы:** jar, war, zip, tar, gzip
- **Java:** javac, java, javadoc, junit
- **Замена текста:** replace, replaceregexp
- **Сеть/деплой:** ftp, telnet, mail
- **Система контроля версий:** cvs, vss
- **Тестирование:** junit, testlet
- **Звук:** sound (воспроизведение музыки!)

### Цели (Targets) и зависимости
```xml
<target name="test" depends="build">
    <!-- сначала выполнится build, потом test -->
</target>
```
`depends="build"` означает: перед выполнением этой цели сначала выполни `build`.

## Maven

**Maven** — более современная система сборки, **декларативный** подход.

Ты говоришь ЧТО хочешь (собрать jar, запустить тесты), а Maven сам знает КАК это сделать.

Управляется файлом `pom.xml` — **Project Object Model**.

### Что описывает pom.xml:
- Имя, версия, тип проекта
- Местонахождение исходного кода
- Зависимости (библиотеки)
- Плагины
- Профили (альтернативные конфигурации)

### Система наименования GAV:
```
groupId : artifactId : version
```
Пример: `javax.portlet:portlet-api:2.0`

В репозитории это хранится как:
```
javax.portlet/portlet-api/2.0/portlet-api-2.0.jar
```

### Зависимости в Maven:
```xml
<dependencies>
    <dependency>
        <groupId>javax.servlet</groupId>
        <artifactId>servlet-api</artifactId>
        <version>2.5</version>
        <scope>provided</scope>  <!-- не включать в архив, берём из сервера -->
    </dependency>
</dependencies>
```

**Scope (область действия) зависимости:**
- `compile` — нужна при компиляции и запуске (по умолчанию)
- `provided` — берётся из контейнера (WildFly, Tomcat), не включается в WAR
- `runtime` — только для запуска, не для компиляции
- `test` — только для тестов

**Транзитивные зависимости:** если твой проект зависит от библиотеки A, которая зависит от B — Maven автоматически скачает и B.

### Репозитории Maven:
1. Maven читает `pom.xml`
2. Скачивает зависимости с удалённого репозитория (`repo1.maven.org`)
3. Сохраняет в локальный репозиторий (`~/.m2/repository`)
4. Использует локальный кэш при следующих сборках

### Жизненный цикл Maven (порядок фаз):
1. `generate-sources` — генерация исходников
2. `compile` — компиляция
3. `test-compile` — компиляция тестов
4. `test` — запуск тестов
5. `package` — упаковка в jar/war
6. `integration-test` — интеграционные тесты
7. `install` — установка в локальный репозиторий
8. `deploy` — деплой на сервер

### Плагины Maven:
Все операции выполняются через плагины:
- **Core:** clean, compiler, deploy, install, resources, surefire
- **Packaging:** jar, war, ear, ejb
- **Reporting, Tools, Thirdparty**

В Ant нужно всё прописывать явно, в Maven достаточно декларативных высказываний — плагины знают что делать.

## Gradle

**Gradle** — современная система сборки, объединяет лучшее из Ant и Maven.

**Преимущества:**
- Нейтрален к языкам программирования (не только Java)
- Базируется на Apache Ant + Ivy
- Использует плагины и жизненный цикл (как Maven)
- Описание зависимостей от Maven (GAV)
- Скрипт сборки на **DSL (Groovy)** — читабельнее XML
- **Инкрементальная и параллельная сборка** — не пересобирает то, что не изменилось

```groovy
apply plugin: "java"
apply plugin: "application"

mainClassName = "ru.ifmo.cs.BcompNG"

repositories {
    mavenCentral()
}

dependencies {
    compile "log4j:log4j-core:2.12.1"
}

jar {
    manifest.attributes("Main-Class": mainClassName)
}
```

## Ivy — менеджер зависимостей для Ant

**Проблема Ant:** нет встроенного управления зависимостями, нужно вручную скачивать JAR-файлы.

**Ivy решает это:** умеет работать с репозиториями Maven2, скачивает зависимости автоматически.

```xml
<!-- в build.xml -->
<target name="resolve">
    <ivy:retrieve/>
</target>
```

```xml
<!-- ivy.xml -->
<dependencies>
    <dependency org="javax.servlet" name="servlet-api" rev="2.5"/>
</dependencies>
```

## GNU autotools — для проектов вне Java

Используется для C/C++ проектов. Решает проблему различий архитектур.

Пользователю достаточно знать три команды:
```bash
./configure    # определяет конфигурацию системы
make           # компилирует
sudo make install  # устанавливает
```

Autotools сами определяют архитектуру, находят библиотеки, генерируют правильный Makefile.

## Сервер сборки / Непрерывная интеграция (CI)

**CI-сервер** автоматически запускает сборку при каждом обновлении кода в репозитории.

**Что делает CI-сервер:**
- Скачивает новый код
- Собирает проект
- Запускает тесты
- Создаёт отчёты
- Хранит все версии сборок

**Примеры CI-серверов:** Jenkins, Travis CI, GitLab CI, CruiseControl.

**Зачем нужно:** если разработчик сломал что-то своим коммитом, CI сразу это обнаружит и уведомит команду.

# ЧАСТЬ 2: ТЕСТИРОВАНИЕ ПО

## Термины тестирования (ISTQB)

| Термин | Определение | Пример |
|--------|-------------|--------|
| **Mistake (Error)** | Ошибка человека, просчёт разработчика | Написал `i=1` вместо `i=0` |
| **Fault** | Дефект в коде — результат ошибки человека | Неверный алгоритм в программе |
| **Failure** | Внешнее проявление дефекта, сбой программы | Программа упала или выдала неверный результат |
| **Error** | Невозможность выполнить задачу вследствие отказа | Пользователь не смог сохранить файл |
| **BUG** | Неформальный термин, означает любое из вышеперечисленного | — |

**Аналогия из жизни:** ты выбрал не того человека в партнёры (mistake) → брак содержит изъяны (fault) → постоянные ссоры (failure) → невозможность быть счастливым (error).

**Важно:** отказы могут быть следствием окружающей среды, а не только кода!

---

## Цели тестирования (по ISTQB)

1. **Обнаружение дефектов** — найти баги до того как их найдёт пользователь
2. **Повышение уверенности** в качестве продукта
3. **Предоставление информации** для принятия решений (выпускать или нет)
4. **Предотвращение дефектов** — на ранних стадиях дешевле исправлять

**Главная цель:** увеличить доверие пользователя к тому, что программа работает корректно во всех необходимых обстоятельствах.

**Важно понимать:** тестирование **не гарантирует** полное отсутствие ошибок — оно лишь повышает уверенность.

---

## Полное тестовое покрытие — почему оно невозможно?

Пример: функция `multiply(int A, int B)`.
- Оба параметра типа `int` = 2^32 значений каждый
- Для полного покрытия нужно 2^64 тестов
- На 3 ГГц процессоре это займёт ~**181,5 лет**

**Вывод:** полное тестовое покрытие недостижимо. Нужно выбирать специфические значения для тестирования.

---

## Статическое и динамическое тестирование

### Статическое тестирование (рецензирование)
- **Не требует запуска кода**
- Можно проводить ещё до написания кода (на спецификациях, требованиях)
- Виды: неформальное ревью, технический анализ, инспекции
- **Цена ошибки растёт с фактором 10** от стадии к стадии — поэтому лучше находить баги раньше

### Динамическое тестирование
- **Требует запуска кода**
- Запуск модулей, групп модулей или всей системы
- При TDD (Test-Driven Development) тесты пишут **до** кода

---

## Методы тестирования: Чёрный и Белый ящик

### Метод «чёрного ящика» (Black Box)
- Внутренняя реализация скрыта
- Тестируем по спецификации/требованиям
- Подаём входные данные → сравниваем с ожидаемым результатом
- **Пример:** тестируем кнопку "Войти" — вводим логин/пароль и проверяем что происходит, не зная как работает авторизация внутри

### Метод «белого ящика» (White Box)
- Можно видеть и анализировать исходный код
- Анализируем пути выполнения, ветвления, условия
- Вычисляем **цикломатическую сложность** — количество путей в программе → это минимальное число тестов
- **Пример:** видим что в коде есть `if/else` → пишем тест для каждой ветки

---

## Тестовый случай (Test Case)

**Тестовый случай** = набор входных данных + ожидаемый результат.

Состоит из:
- **Входные значения** — данные или управляющие воздействия
- **Предусловия** — состояние системы перед тестом
- **Ожидаемый результат** — что должно произойти
- **Постусловия** — состояние после теста

**Пример тестового случая для AreaCheck:**
- Входные данные: x=0.5, y=0.5, r=1.0
- Предусловие: создан объект AreaCheck
- Ожидаемый результат: `checkHit()` возвращает `true`
- Постусловие: состояние не изменилось

**Важно:** ожидаемый результат определяется **до** запуска теста!

---

## Тестовый сценарий (Test Scenario)

**Тестовый сценарий** = последовательность тестовых случаев.

**Пример — банкомат:**

| №  | Нач. состояние | Ввод | Действие системы | Вывод | Кон. состояние |
|----|---------------|------|-----------------|-------|----------------|
| 1  | Готов | Вставить карту | Чтение карты | "Введите PIN" | Ожидание PIN |
| 2  | Ожидание PIN | Верный PIN | Проверка PIN | Выбор транзакции | Ожидание выбора |
| 3  | Ожидание выбора | Снять 5000р | Проверка баланса | Деньги | Выдача денег |
| 4  | Выдача денег | Забрать деньги и карту | Завершение | Благодарность | Готов |

**Сценарии должны охватывать:** и нормальное поведение, и ошибочные ситуации!

## Выбор тестового покрытия

### Эквивалентное разбиение (Partition Equivalence)
Разбиваем входные данные на **классы эквивалентности** — группы, где программа ведёт себя одинаково. Тестируем по одному значению из каждой группы.

**Пример для нашего AreaCheck:**
- Первый квадрант (x≥0, y≥0) — одна четверть круга
- Второй квадрант (x≤0, y≥0) — прямоугольник
- Третий квадрант (x<0, y<0) — всегда промах
- Четвёртый квадрант (x≥0, y≤0) — треугольник

Для каждого класса тестируем: точку внутри области, снаружи, и граничное значение.

### Анализ граничных значений
Ошибки чаще всего встречаются на границах. Тестируем значения прямо на границе, чуть меньше и чуть больше.

**Пример:** если y должен быть в [-3, 3] — тестируем y=-3, y=0, y=3, y=-3.1, y=3.1.

### Таблица решений
Составляем таблицу всех комбинаций входных условий и ожидаемых результатов.

### Таблица переходов
Описываем состояния системы и переходы между ними — покрываем каждый переход тестом.

## Модульное (юнит) тестирование

**Модуль** — отдельный компонент: метод, класс, или программный модуль.

**Цель:** протестировать компонент **изолированно** от остальной системы.

**Изолирование модулей:**
- **Драйвер** — заменяет вызывающий модуль (тот кто вызывает тестируемый)
- **Заглушка (Stub)** — заменяет подчинённый модуль (тот кого вызывает тестируемый), возвращает заранее определённые значения

**Пример:** тестируем `AreaCheck` — он не зависит от других классов, поэтому тестируем напрямую. Если бы он зависел от БД — создали бы заглушку вместо реальной БД.

## JUnit — фреймворк для модульного тестирования Java

### Основные аннотации:

```java
@Test               // метод является тестом
@Before             // выполняется перед каждым тестом
@After              // выполняется после каждого теста
@BeforeClass        // выполняется один раз перед всеми тестами класса
@AfterClass         // выполняется один раз после всех тестов класса
```

### Методы проверки (Assert):

```java
assertEquals(expected, actual)      // проверка равенства
assertTrue(condition)               // проверка что условие истинно
assertFalse(condition)              // проверка что условие ложно
assertNull(object)                  // проверка что объект null
assertNotNull(object)               // проверка что объект не null
assertArrayEquals(expected, actual) // проверка равенства массивов
```

### Пример теста из нашей лабы:

```java
public class AreaCheckTest {
    private final AreaCheck areaCheck = new AreaCheck();

    @Test
    public void testHitInCircleQ1() {
        // Точка (0.5, 0.5) при R=1 попадает в четверть круга
        assertTrue(areaCheck.checkHit(0.5, 0.5, 1.0));
    }

    @Test
    public void testMissQ3() {
        // В третьем квадранте всегда промах
        assertFalse(areaCheck.checkHit(-0.5, -0.5, 1.0));
    }
}
```

**Как работает JUnit:**
1. Находит все методы с аннотацией `@Test` через Reflection API
2. Для каждого теста создаёт новый экземпляр класса
3. Запускает `@Before`, затем тест, затем `@After`
4. Записывает результат в журнал
5. Порядок тестов не гарантирован — они независимы друг от друга

## Интеграционное тестирование

**Цель:** проверить **взаимодействие** между модулями.

**Что проверяет:**
- Вызовы API между компонентами
- Работу с базой данных
- Пользовательский интерфейс
- Сетевые взаимодействия

**Стратегии интеграции:**

### Сверху вниз (Top-Down)
- Сначала тестируем бизнес-логику верхнего уровня
- Подчинённые модули заменяем заглушками
- Потом постепенно подключаем реальные модули снизу
- **Плюс:** быстро появляется "осязаемое" приложение
- **Минус:** много заглушек

### Снизу вверх (Bottom-Up)
- Начинаем с модулей нижнего уровня (БД, аппаратура)
- Постепенно подключаем верхние слои
- Используем Драйверы вместо вызывающих модулей
- Хорошо для систем тесно связанных с аппаратурой

### Функциональная (End-to-End)
- Интегрируем по функциям — полный путь "UI → Логика → БД"
- Добавляем функцию за функцией

### Большой взрыв (Big Bang)
- Собираем всё сразу
- **Минус:** при ошибках непонятно где именно проблема

## Функциональное тестирование

- Проверяет функциональность системы **с точки зрения пользователя**
- На базе сценариев использования
- Начинается с UI (интерфейса пользователя)
- Инструменты автоматизации: **Selenium**, Sahi, Watir

---

## Тестирование системы в целом

Начинается после завершения интеграции. Проходит несколько фаз (от простого к сложному):

1. **Возможности** — проверяем заявленный функционал
2. **Стабильность** — несколько пользователей одновременно
3. **Устойчивость к сбоям** — вводим заведомо неверные данные
4. **Совместимость** — кросс-браузерность, совместимость с другим ПО
5. **Производительность** — нагрузочное тестирование

**Этапы тестирования системы:**
- **Системное тестирование** — внутри организации-разработчика
- **Альфа-тестирование** — пользователь под контролем разработчика
- **Бета-тестирование** — пользователь в реальном окружении
- **Приёмочное тестирование** — пользователь в своём окружении без контроля (решает: платить или нет)

## Производительность — CARAT

Тестирование производительности включает:

| Буква | Термин | Что означает |
|-------|--------|--------------|
| **C** | Capacity | Нефункциональные возможности (объём обрабатываемых данных) |
| **A** | Accuracy | Точность математических расчётов |
| **R** | Response Time | Время ответа системы на запрос (норма: 1-5 секунд) |
| **A** | Availability | Готовность системы: `(MTBF - MTTR) / MTBF` |
| **T** | Throughput | Пропускная способность (запросов в секунду) |

**Availability пример:** коэффициент 0.99999 = система может не работать всего 5 минут в год.

**Инструменты нагрузочного тестирования:** HP Load Runner, Apache JMeter, The Grinder, Tsung.

## V-модель тестирования

Каждому этапу разработки соответствует свой уровень тестирования:

```
Требования          ←→    Приёмочное тестирование
  Анализ            ←→    Системное тестирование
    Архитектура     ←→    Интеграционное тестирование
      Детальное проектирование ←→ Модульное тестирование
              Разработка
```

- **Валидация** — "Have we done the right thing?" — Правильно ли определили что делать?
- **Верификация** — "Have we done the thing right?" — Правильно ли сделали?

## Автоматизация тестов

**Зачем:**
- Регрессионное тестирование — при изменениях запускаем старые тесты
- Повторяемость — одинаковые входные данные → одинаковый результат
- Проверка в разных окружениях (совместимость)

**Когда не нужна автоматизация:**
- Простые задачи, где проще нанять тестировщика
- Если функционал часто меняется (тесты устаревают)

## Статические методы тестирования

Три основных метода:

| | Сквозной контроль | Технический анализ | Инспекция |
|--|------------------|-------------------|-----------|
| **Цель** | Поиск дефектов + обмен опытом | Поиск дефектов + принятие решений | Поиск дефектов + улучшение процесса |
| **Ведущий** | Автор | По обстоятельствам | Подготовленный модератор |
| **Группа** | 2-7 человек | >3 | 3-6 |
| **Формальность** | Обычно нет | Иногда | Всегда |
| **Отчёт** | Неформальный | Формальный | Формальный + метрики |

## 1. Тестирование ПО. Цель тестирования, виды тестирования.

**Тестирование ПО** — процесс проверки программного обеспечения с целью обнаружения дефектов и повышения уверенности в его качестве.

**Цели тестирования (ISTQB):**
- Обнаружение дефектов
- Повышение уверенности в уровне качества
- Предоставление информации для принятия решений
- Предотвращение дефектов

**Виды тестирования:**

По **методу:**
- Статическое (без запуска кода — ревью, инспекции)
- Динамическое (с запуском кода)

По **уровню:**
- Модульное (юнит) — тестирование отдельных классов/методов
- Интеграционное — тестирование взаимодействия модулей
- Системное — тестирование всей системы
- Приёмочное — пользователь проверяет систему

По **подходу:**
- "Чёрного ящика" — без знания внутренней реализации
- "Белого ящика" — с анализом исходного кода

По **характеру:**
- Функциональное — проверка функций по сценариям
- Нагрузочное (производительность) — CARAT
- Регрессионное — проверка что изменения не сломали старое

## 2. Модульное тестирование, основные принципы и используемые подходы.

**Модульное тестирование** — тестирование отдельных компонентов ПО в изоляции от остальной системы (IEEE 610).

**Основные принципы:**
1. **Изоляция** — тестируем модуль независимо от других
2. **Повторяемость** — одинаковые входные данные дают одинаковый результат
3. **Независимость** — тесты не зависят друг от друга
4. **Быстрота** — юнит-тесты должны выполняться быстро

**Как изолировать модуль:**
- **Драйвер** — заменяет вызывающий модуль (управляет тестированием)
- **Заглушка (Stub)** — заменяет зависимость, возвращает заранее определённые значения

**Подходы:**
- **TDD (Test-Driven Development)** — сначала пишем тест, потом код. Тест сначала падает, по мере написания кода "зеленеет"
- **Классический подход** — сначала код, потом тесты
- **Метод белого ящика** — анализируем все ветки кода и пишем тесты для каждой

**Выбор тестовых данных:**
- Эквивалентное разбиение — по одному тесту из каждого класса эквивалентности
- Граничные значения — тестируем края допустимых диапазонов
- Граничные случаи — null, 0, пустая строка, отрицательные числа

## 3. Пакет JUnit, основные API.

**JUnit** — фреймворк для написания и запуска модульных тестов в Java.

**Как работает:**
1. Ищет классы с тестами через аннотации (Reflection API)
2. Для каждого `@Test` метода создаёт новый экземпляр класса
3. Выполняет `@Before`, `@Test`, `@After`
4. Записывает результаты

**Основные аннотации JUnit 4:**
```java
@Test                    // тестовый метод
@Before                  // перед каждым тестом
@After                   // после каждого теста
@BeforeClass             // один раз перед всеми тестами
@AfterClass              // один раз после всех тестов
@Ignore                  // пропустить тест
@Test(expected=Exception.class)  // ожидаем исключение
@Test(timeout=1000)      // тест должен завершиться за 1000мс
```

**Основные методы Assert:**
```java
// Равенство
assertEquals(100, result);
assertEquals("hello", str);
assertEquals(3.14, pi, 0.001);  // для double — с погрешностью

// Булевы
assertTrue(list.isEmpty());
assertFalse(user.isAdmin());

// Null
assertNull(result);
assertNotNull(connection);

// Массивы
assertArrayEquals(new int[]{1,2,3}, array);

// Провал теста
fail("Этот код не должен выполняться");
```

**Запуск в Ant:**
```xml
<junit printsummary="yes" fork="true">
    <classpath refid="test.classpath"/>
    <formatter type="plain"/>
    <batchtest todir="build/test-reports">
        <fileset dir="build/test-classes" includes="**/*Test.class"/>
    </batchtest>
</junit>
```

## 4. Системы автоматической сборки. Назначение, принципы работы, примеры систем.

**Назначение:** для автоматизации рутинных операций превращения исходного кода в готовый программный продукт — компиляция, тестирование, упаковка в архивы, деплой.

**Принципы работы:**
- Описание сборки в специальном файле (build.xml, pom.xml, build.gradle)
- Определение зависимостей между шагами (target depends)
- Инкрементальная сборка — пересборка только изменённых файлов
- Управление зависимостями от внешних библиотек

**Примеры систем:**

| Система | Язык описания | Подход | Главный файл |
|---------|--------------|--------|-------------|
| **Make** | Makefile | Императивный | Makefile |
| **Ant** | XML | Императивный | build.xml |
| **Maven** | XML | Декларативный | pom.xml |
| **Gradle** | Groovy DSL | Смешанный | build.gradle |

**Make** — первопроходец, для Unix/C++. Учитывает время модификации файлов.

**Ant** — для Java, XML-описание, много встроенных команд, но нет управления зависимостями.

**Maven** — декларативный, управляет зависимостями через репозитории, жизненный цикл.

**Gradle** — современный, быстрый, инкрементальная сборка, используется в Android и крупных Java-проектах.

## 5. Утилита Make. Make-файлы, цели и правила.

**Make** — утилита автоматизации сборки, появилась в Unix.

**Makefile** — файл с описанием правил сборки.

**Структура правила:**
```makefile
цель: зависимости
    команда    # ВАЖНО: отступ должен быть ТАБОМ, не пробелами!
```

**Пример:**
```makefile
# Переменные
CC = gcc
CFLAGS = -O -g
OBJS = main.o utils.o

# Цель по умолчанию
all: myapp

# Правило сборки
myapp: $(OBJS)
    $(CC) $(CFLAGS) -o myapp $(OBJS)

# Правило компиляции
main.o: main.c utils.h
    $(CC) $(CFLAGS) -c main.c

utils.o: utils.c utils.h
    $(CC) $(CFLAGS) -c utils.c

# Очистка
clean:
    rm -f myapp $(OBJS)
```

**Как работает Make:**
1. Читает Makefile, начинает с цели `all`
2. Проверяет зависимости — нужно ли пересобирать
3. Сравнивает время модификации цели и её зависимостей
4. Если зависимость новее цели — пересобирает
5. Выполняет команды в правильном порядке

**Цели Make:**
- Обычные цели — создают файлы
- Псевдоцели (phony) — не создают файлы (clean, all, install)

**Недостатки Make:** платформозависимый, сложный синтаксис, много неявных правил.

## 6. Утилита Ant. Сценарии сборки, цели и команды.

**Ant (Apache Ant)** — инструмент автоматизации сборки Java-проектов.

**Управляется файлом `build.xml`:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project name="myproject" default="build" basedir=".">

    <!-- Загрузка свойств из файла -->
    <property file="build.properties"/>

    <!-- Classpath -->
    <path id="compile.classpath">
        <fileset dir="lib" includes="**/*.jar"/>
    </path>

    <!-- Цель: компиляция -->
    <target name="compile">
        <mkdir dir="build/classes"/>
        <javac srcdir="src" destdir="build/classes"
               includeantruntime="false">
            <classpath refid="compile.classpath"/>
        </javac>
    </target>

    <!-- Цель: сборка JAR (depends = сначала выполни compile) -->
    <target name="build" depends="compile">
        <jar destfile="build/myapp.jar" basedir="build/classes">
            <manifest>
                <attribute name="Main-Class" value="com.example.Main"/>
            </manifest>
        </jar>
    </target>

    <!-- Цель: очистка -->
    <target name="clean">
        <delete dir="build"/>
    </target>

    <!-- Цель: тесты -->
    <target name="test" depends="build">
        <junit printsummary="yes">
            <classpath refid="compile.classpath"/>
            <batchtest>
                <fileset dir="build/classes" includes="**/*Test.class"/>
            </batchtest>
        </junit>
    </target>

    <!-- Цель: музыка -->
    <target name="music" depends="build">
        <sound>
            <success source="audio/success.wav"/>
        </sound>
    </target>

    <!-- Цель: alt — замена имён -->
    <target name="alt">
        <copy todir="build/alt-src">
            <fileset dir="src"/>
        </copy>
        <replaceregexp match="OldClass" replace="NewClass" flags="g">
            <fileset dir="build/alt-src" includes="**/*.java"/>
        </replaceregexp>
        <javac srcdir="build/alt-src" destdir="build/alt-classes"/>
        <jar destfile="build/myapp-alt.jar" basedir="build/alt-classes"/>
    </target>

</project>
```

**Файл свойств `build.properties`:**
```properties
src.dir=src/main/java
build.dir=build
jar.file=build/myapp.jar
main.class=com.example.Main
project.version=1.0
```

**Основные команды (tasks) Ant:**

| Команда | Что делает |
|---------|-----------|
| `<javac>` | Компиляция Java-кода |
| `<jar>` | Создание JAR-архива |
| `<war>` | Создание WAR-архива |
| `<mkdir>` | Создание директории |
| `<delete>` | Удаление файлов/папок |
| `<copy>` | Копирование файлов |
| `<move>` | Перемещение файлов |
| `<echo>` | Вывод сообщения |
| `<junit>` | Запуск JUnit-тестов |
| `<replace>` | Замена текста в файлах |
| `<replaceregexp>` | Замена по регулярному выражению |
| `<sound>` | Воспроизведение звука |
| `<property>` | Установка свойства/переменной |
| `<path>` | Определение classpath |

**Запуск Ant:**
```cmd
ant               # выполняет цель по умолчанию (default)
ant compile       # выполняет цель compile
ant build         # выполняет цель build (и compile как зависимость)
ant clean build   # последовательно: clean, затем build
```

**Важные особенности Ant:**
- Свойства **неизменяемы** — нельзя переопределить после установки
- `depends` — гарантирует порядок выполнения
- MANIFEST.MF должна содержать `Main-Class` и версию
- Все переменные выносятся в `build.properties`
- Каждый этап — отдельный `target`

#### 静默编译 silent-compile
При компиляции исходного кода компьютера эта функция маскирует или скрывает подробную информацию о выходных данных, журналы и предупреждения в процессе компиляции, отображая на терминале или в интерфейсе только окончательный результат сборки или отчет об ошибках.  
在编译计算机源代码时，屏蔽或隐藏编译过程中的详细输出信息、日志以及警告（Warnings），只在终端或界面上显示最终构建结果或错误报告。  

<macrodef> 是一个内置标签，允许开发者自定义可重用的任务（Task）或标签`<macrodef>` — это встроенный тег, позволяющий разработчикам настраивать многократно используемые задачи или теги.

#### 自动装配系统两类
1. Задачно-ориентированные — 基于任务型
RU: Вы описываете как собирать — явная последовательность задач, разработчик контролирует порядок и логику вручную.
CN: 你定义如何构建 —— 显式的任务序列，开发者手动控制顺序和逻辑。
`Examples: Ant, Make, Rake`

2. Convention-based — Основанные на соглашениях — 基于约定型
RU: Вы описываете что хотите получить — инструмент сам определяет порядок и способ сборки на основе стандартных соглашений.
CN: 你定义想要什么 —— 工具自动决定如何构建，基于标准项目结构约定。
`Examples: Maven, Gradle`

##### Ключевые различия / 核心区别 / Key Differences
RU:

Задачно-ориентированные системы дают полный контроль, но требуют описания каждого шага вручную. При росте проекта build.xml становится громоздким и трудно поддерживаемым.
Основанные на соглашениях системы работают по принципу «соглашение важнее конфигурации» — если вы следуете стандартной структуре, конфигурации почти не нужно. Но отклонение от стандарта требует дополнительных усилий.

CN:

基于任务型系统给予完全控制权，但每一步都需要手动描述。随着项目增长，build.xml 会变得臃肿难以维护。
基于约定型系统遵循*"约定优于配置"*原则 —— 只要遵循标准结构，几乎不需要配置。但偏离标准时需要额外工作量。

Особенности каждого / 各自特点 / Characteristics
Task-based 基于任务型
RU:

Полная свобода в организации сборки
Нет встроенного управления зависимостями
Легко читается для простых проектов
Сложно масштабировать

CN:

构建组织完全自由
没有内置依赖管理
简单项目易读易懂
难以规模化扩展

Convention-based 基于约定型
RU:

Автоматическое управление зависимостями (Maven Central и др.)
Жёсткая структура каталогов (src/main/java, src/test/java)
Меньше кода конфигурации для стандартных проектов
Встроенный жизненный цикл сборки (compile → test → package → deploy)

CN:

自动依赖管理（Maven Central 等）
强制目录结构（src/main/java、src/test/java）
标准项目配置量极少
内置构建生命周期（compile → test → package → deploy）
