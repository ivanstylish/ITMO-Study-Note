# Wrench ISA 实验问题解答

以下是对计算机架构课程中关于 Wrench 模拟器（Acc32、F32a、M68k、RISC-IV、VLIW-IV 等 ISA）所有问题的详细解答。答案结合实验手册内容、ISA 文档及实际编程经验组织，长度适中，便于理解和报告撰写。

## 1. 什么是标签（label）？/ Что такое метка (label)?

**中文**：标签（label）是汇编语言中用于标记内存地址的符号名称。它以 `名称:` 的形式定义，可以指向数据（.data 节）或代码（.text 节）。标签使程序更具可读性，避免手动计算硬编码地址。

**俄文**：Метка (label) — это символическое имя, которое ассемблер связывает с конкретным адресом в памяти. Определяется как `имя:`. Используется для ссылки на данные или инструкции, делая код читаемым и облегчая перемещение секций.

在 Wrench 中，所有标签必须唯一，汇编器会自动解析并分配最终地址。

## 2. 标签的值从哪里计算？/ Откуда рассчитываются значения меток (label)?

**中文**：标签的值相对于其所在节（.data 或 .text）的起始地址计算。如果使用 `.org 地址` 指令，则从指定的绝对地址开始计算；否则，汇编器从 0x0000 开始顺序放置各节并合并同类型节，最后统一计算所有标签地址。

**俄文**：Значения меток рассчитываются относительно начала секции (.data или .text). При использовании `.org адрес` метка привязывается к указанному абсолютному адресу. Без `.org` секции размещаются последовательно с 0x0000.

## 3. 对齐（alignment，不使用的缓冲区在数据节中）的用途是什么？/ Назначение alignment (неиспользуемый буфер в секции данных)?

**中文**：对齐（alignment）用于确保数据（如 32 位字）从地址的整数倍位置开始存放（通常 4 字节对齐）。未使用的缓冲区（padding）会自动或手动插入，以避免跨字访问错误，提高访问效率和硬件兼容性。

**俄文**：Alignment обеспечивает выравнивание данных по границам (например, слова по 4 байта). Неиспользуемый буфер (padding) вставляется для предотвращения ошибок выравнивания и повышения производительности.

## 4. 在 Acc32 中，为什么定义了重复的指令 load_addr 和 load？/ acc32 Зачем определены дублирующиеся инструкции load_addr и load?

**中文**：`load <offset>` 是相对当前 PC 的相对加载（适合位置无关代码和循环），而 `load_addr <address>` 是绝对地址加载。两者并存便于程序员根据场景选择：相对寻址更灵活，绝对寻址更直观。这是累加器架构中简化教学不同寻址模式的常见设计。

**俄文**：`load <offset>` — относительная загрузка от PC, `load_addr <address>` — абсолютная. Дублирование позволяет выбирать удобный режим адресации: относительный для позиционно-независимого кода, абсолютный для глобальных данных.

## 5. Acc32 中的溢出和进位。Overflow 和 Carry 标志，以及其他处理方式 / acc32 Переполнение и перенос. Overflow и Carry флаги, иные способы обработки.

**中文**：在 Acc32 中，Carry (C) 标志表示无符号运算产生进位，Overflow (V) 标志表示有符号运算结果符号错误（溢出）。两者主要在 `add`、`sub` 等算术指令中更新。其他处理方式包括：使用分支指令 `bvs`/`bvc`、`bcs`/`bcc` 判断标志，或通过 `clv`/`clc` 清除标志。对于无符号运算可忽略 Overflow。

**俄文**：Carry (C) — перенос при беззнаковых операциях, Overflow (V) — переполнение при знаковых. Устанавливаются в арифметических инструкциях. Другие способы: условные переходы (bvs/bvc, bcs/bcc) или очистка флагов (clv/clc).

## 6. 在 Acc32 中，可以删除哪些 Load* 操作以保留处理器能力？/ acc32 Какие Load* операции можно выкинуть чтобы сохранить возможности процессора?

**中文**：可以删除 `load`（相对）或 `load_addr`（绝对）中的一个，保留 `load_imm` 和 `load_acc`（间接通过 Acc）。通过 `load_imm` + 算术运算 + `load_acc` 仍能模拟绝对和相对加载，保持处理器图灵完备性。但会增加指令条数，降低代码简洁性。

**俄文**：Можно удалить либо `load`, либо `load_addr`, оставив `load_imm` и `load_acc`. Комбинация позволяет эмулировать остальные режимы загрузки, сохраняя функциональность процессора.

## 7. 什么是内存映射 I/O（Memory Mapped IO）？/ Что такое Memory Mapped IO?

**中文**：内存映射 I/O 是将输入输出设备映射到内存地址空间的技术。处理器使用普通的 load/store 指令访问固定地址（如 0x80）来读写外设，而不需要专门的 I/O 指令。在 Wrench 中通过配置文件 `.yaml` 中的 `memory_mapped_io` 进行配置，极大简化了体系结构设计。

**俄文**：Memory Mapped I/O — способ адресации устройств ввода-вывода как обычной памяти. Доступ осуществляется обычными инструкциями load/store по фиксированным адресам.

## 8. 如果处理器尝试从 I/O 地址读取指令，会发生什么？/ Что будет, если процессор попытается прочитать инструкцию из адреса ввода-вывода?

**中文**：行为取决于具体实现。在 Wrench 中，通常会读取当前 I/O 端口的值并将其作为指令解释，但由于 opcode 很可能无效，可能导致非法指令异常或程序意外终止。建议使用 `.org` 将代码段与 I/O 地址空间严格分离，避免此类情况。

**俄文**：Процессор может прочитать значение порта как данные, но интерпретация как инструкции обычно приводит к некорректному opcode или исключению. Рекомендуется отделять код от MMIO.

## 9. 指令...是什么意思？解释...行中发生的事情 / Что означает инструкция ...? Объясните что происходит в строках ....

**中文**：（本题为占位符，请替换为具体指令）以 Acc32 的 `add <address>` 为例：该指令将累加器 Acc 的值与内存地址中的值相加，结果存回 Acc，并更新 Carry 和 Overflow 标志。执行过程包括：取指令、取操作数、ALU 计算、写回结果、更新状态标志。

**俄文**：(Замените на конкретную инструкцию). Например, `add <address>` в Acc32: складывает Acc с mem[`<address>`], результат в Acc, обновляет флаги C и V.

## 10. 如何减少机器码体积？会带来哪些限制/问题？/ Как сократить объём машинного кода? Какие ограничения/проблемы это создаст?

**中文**：减少机器码体积的方法包括：优先使用相对寻址、公共子程序、短立即数优化（如 RISC-IV 中的 %hi/%lo）、消除冗余的 load/store 操作。  
**可能的问题**：代码可读性下降、调试难度增加、偏移量溢出风险、因指令条数增多导致执行周期变长。

**俄文**：Способы: относительная адресация, общие подпрограммы, %hi/%lo. Проблемы: снижение читаемости, сложность отладки, риск переполнения смещений.

## 11. 在 RISC-IV 中，0(t1) 结构是什么意思？为什么需要它？/ risc-iv Что означает конструкция 0(t1)? Зачем она нужна?

**中文**：`0(t1)` 表示基址寻址模式，其中偏移量为 0，基址寄存器为 t1。实际访问地址 = t1 的内容 + 0。主要用于通过寄存器保存的指针进行间接内存访问（如数组、栈、动态数据结构）。

**俄文**：`0(t1)` — базовая адресация с нулевым смещением. Нужна для косвенного доступа по адресу, хранящемуся в регистре (указатели, стек и т.д.).

## 12. 在 RISC-IV 中，为什么加载一个字到寄存器需要两条指令？/ risc-iv Почему для загрузки слова регистр нужно две инструкции?

**中文**：因为 RISC-IV 指令长度固定为 4 字节，立即数字段有限（addi 只有 12 位）。加载 32 位地址需先用 `lui` 加载高 20 位，再用 `addi` 加上低 12 位（配合 %hi 和 %lo 伪指令）。这种设计保持了指令集的简洁性和正交性。

**俄文**：Из-за фиксированной длины инструкции и ограничения immediate (12 бит). Полный 32-битный адрес загружается через `lui` (верхние 20 бит) + `addi` (нижние 12 бит).

## 13. 在 F32a 中，为什么使用过程（子程序）很方便？/ f32a Почему в F32a удобно использовать процедуры?

**中文**：F32a 是栈式架构（类似 Forth）。调用子程序时会自动将返回地址压入返回栈（return stack），返回指令 `;` 自动弹出并恢复 PC。参数和局部变量也通过数据栈管理，无需手动保存大量寄存器，特别适合递归和嵌套调用，代码简洁高效。

**俄文**：F32a — стековая архитектура. Вызов автоматически сохраняет адрес возврата в return stack, возврат — `;`. Удобно для рекурсии и передачи параметров через стек.

## 14. F32a 中的扩展算术模式（Extended Arithmetic Mode）。用途和使用 / f32a Extended arithmetic mode. Назначение и использование.

**中文**：Extended Arithmetic Mode (EAM) 允许加法指令 `+` 考虑进位标志（带进位加法）。通过 `eam` 指令从栈中弹出值来启用/禁用该模式。主要用于多精度（大整数）运算和长链式算术操作，避免进位丢失。

**俄文**：EAM позволяет инструкции `+` учитывать carry. Устанавливается командой `eam`. Используется для многословной арифметики (big integer).

## 15. 如何迭代字数组？如何迭代结构数组？/ Как бы вы итерировались по массиву слов? По массиву структур?

**中文**：  
- **字数组**：使用寄存器保存数组基址，通过 `load`/`@` 或 `lw` 取值，每次循环后指针加 4（字长），结合 `bnez` 等分支构成循环。  
- **结构数组**：同样使用基址指针 + 不同偏移量访问结构成员（如 M68k 中的 `8(A0)` 或 RISC-IV 中的 `offset(rs1)`）。

**俄文**：Для массива слов — указатель + инкремент на 4. Для структур — указатель + разные смещения (displacement).

## 16. 指令编码的特点、代码密度、访问问题 / Особенности кодирования инструкций, плотность кода, проблемы доступа.

**中文**：Acc32 和 F32a 采用变长编码，代码密度较高但解码复杂；RISC-IV 和 VLIW-IV 采用定长编码（4 字节 / 11 字节），解码简单但代码密度较低。常见问题包括：地址对齐要求、缓存未命中、立即数/偏移量范围限制等。

**俄文**：Acc32/F32a — переменная длина (высокая плотность), RISC/VLIW — фиксированная (проще декодирование, ниже плотность). Проблемы: выравнивание, ограничения immediate/offset.

## 17. 如何减少机器码体积？请演示 / Как вы можете сократить объём машинного кода? Продемонстрируйте.

**中文**：方法包括使用相对寻址、提取公共子程序、使用 %hi/%lo 加载大立即数、减少冗余的 dup/drop 等。  
**示例**（RISC-IV）：原代码使用多条 `addi` 加载常量 → 优化后用 `lui` + `addi` + `%hi/%lo`，体积可减少 30%~50%。

**俄文**：Использовать относительную адресацию, общие подпрограммы, %hi/%lo. Пример: замена нескольких addi на lui + addi сокращает код.

## 18. RISC-IV 中的栈操作。内存分配、过程间数据传递 / risc-iv Работа со стеком. Выделение памяти, передача данных в/из процедур.

**中文**：栈通常向下增长（SP 寄存器）。分配栈帧使用 `addi sp, sp, -N`。参数可通过寄存器（A0–A7）或压栈传递，返回值通常放在 A0 中。被调用者需手动保存 callee-saved 寄存器（S* 系列）。调用使用 `jal` 指令。

**俄文**：Стек растёт вниз. Выделение: `addi sp, sp, -N`. Параметры — в регистрах A* или на стеке, возврат значения — в A0.

## 19. 在原理图（schema）中，Nop 指令看起来如何？如何执行？/ schema Как будет выглядеть Nop и как он будет выполняться?

**中文**：Nop 通常编码为特殊 opcode（如全零或 `addi x0, x0, 0`）。在原理图中，控制单元解码后不激活 ALU、内存或写回操作，仅将 PC 增加指令长度（4 或 11 字节），不改变任何状态。执行时间为 1 个周期。

**俄文**：Nop — специальный opcode (часто все нули). Control Unit пропускает операции, только PC += размер инструкции.

## 20. Jmp 指令在原理图中如何？如何执行？/ schema Как будет выглядеть Jmp и как он будет выполняться?

**中文**：Jmp 指令包含目标地址或偏移。在原理图中，控制单元通过多路选择器（mux）将新地址直接送入 PC，不保存返回地址。执行流程：取指 → 解码 → 更新 PC → 从新地址取下一条指令。

**俄文**：Jmp загружает новое значение в PC через mux. Нет сохранения возврата.

## 21. Add 指令在原理图中如何？如何执行？/ schema Как будет выглядеть Add и как он будет выполняться?

**中文**：Add 指令涉及取两个操作数送入 ALU 进行加法运算，结果写回目标寄存器（或 Acc），同时更新状态标志。在原理图中路径为：取指 → 解码 → 读寄存器/内存 → ALU → 写回 + 更新标志寄存器。

**俄文**：Операнды → ALU (сложение) → запись результата + обновление флагов C/V.

## 22. 寄存器文件内部如何工作？/ schema Как работает внутри регистровый файл?

**中文**：寄存器文件是由多个快速 SRAM 单元组成的阵列，包含读端口（通常两个）和写端口。地址解码器根据寄存器编号选择对应单元，多路复用器将数据送到数据总线，写操作由写使能（WE）信号控制，支持并行读写多个寄存器。

**俄文**：Массив SRAM с адресными портами чтения и записи. Декодер выбирает регистр, мультиплексоры выводят данные.

## 23. 内存映射 I/O 内部如何工作？/ schema Как работает внутри Memory Mapped I/O?

**中文**：MMIO 在地址解码阶段被识别为 I/O 地址范围。当地址落在 MMIO 区间时，地址译码器激活外设控制器而非普通 RAM，通过数据总线与设备交换数据。外部看起来与内存访问完全一致，无需额外 I/O 指令。

**俄文**：Адрес декодируется в диапазон устройств. Вместо RAM активируется периферия, данные идут через общую шину.

## 24. 控制单元（Control Unit）中是否有寄存器？/ schema Есть ли регистры в Control Unit?

**中文**：是的，控制单元通常包含以下寄存器：指令寄存器（IR，用于保存当前指令）、程序计数器（PC）、状态标志寄存器，以及可能的临时寄存器或微码状态寄存器。控制单元根据这些寄存器和指令 opcode 生成各种控制信号。

**俄文**：Да, обычно присутствуют: Instruction Register (IR), Program Counter (PC), регистры флагов и временные регистры.
