RLE（Run-Length Encoding，游程编码）的本质是：用"数量+符号"两个字节来代替连续重复的多个字节，当重复序列够长时可以压缩数据。
整个程序分三个阶段，用下面这张图来描述：
```
IO 0x80 ──读──→ [input_buf]  ──压缩──→ [output_buf]  ──写──→ IO 0x84

Phase 1: 读      Phase 2: RLE 编码       Phase 3: 输出
```

1. 阶段 1（读输入）：从内存映射 IO 端口 0x80 一次读一个字节存入内存缓冲区，遇到 \n 或读满 64 字节停止。D4 记录读到了多少个有效字符。
2. 阶段 2（核心：RLE 编码）：用"双指针"思路扫描输入缓冲区。

D2 保存当前 run 的字符，D1 保存它连续出现的次数
每次看下一个字符：如果和 D2 相同且 D1 < 9，就让 D1++
否则 flush：把 ('0'+D1) 和 D2 这两个字节写入输出缓冲区，然后用下一个字符开新的 run
上限 9：因为我们用一个 ASCII 数字来表示计数，'1'~'9' 刚好是 9 个状态，超过就必须断开

3. 阶段 3（输出）：从输出缓冲区逐字节读出，写到 IO 端口 0x84，遇到 null 终止。
溢出处理：如果压缩后的长度 + 1（null 终止符）≥ 64，说明输出缓冲区装不下，写单个 0xFF 错误标记。值得注意的是：RLE 对于完全没有重复的字符串（如 "ABC"）实际上会膨胀——3A4B4C 比 AAABBBBCCCC 短，但 1A1B1C 比 ABC 长。这是 RLE 的固有局限性。

1. 算法逻辑与边界情况 (Логика алгоритма и граничные случаи)
问：为什么计数器 D1 的上限是 9？如果输入有 15 个连续的 'A' 会怎么输出？
答：因为输出格式是“数量+字符”，且数量只占用1个字节。1个字节的 ASCII 数字只能是 '1' 到 '9'。如果输入 15 个 'A'，程序会先输出 9A，然后重置计数器，继续统计剩下的 6 个 'A'，输出 6A。最终结果是 9A6A。
问：这种 RLE 压缩算法在什么情况下不仅不能压缩，反而会膨胀（变长）？
答：当输入没有连续相同字符时，例如输入 ABC，输出会变成 1A1B1C。原本 3 字节的数据变成了 6 字节，体积翻倍。这是基础 RLE 算法的典型缺陷。
问：如果输入为空（直接输入换行符），程序会怎样？
答：在 read_done 标签处，程序会检查 D4 是否为 0。如果为 0，程序会直接执行 halt 停机，不会进入压缩阶段，也不会有任何输出。
2. M68k 架构与指令细节 (Детали архитектуры M68k и инструкций)
问：指令 move.b (A0)+, D2 中的 (A0)+ 是什么意思？
答：这是地址寄存器间接寻址带后增量。它先使用 A0 中存储的地址读取内存数据到 D2，然后根据操作大小（这里是 .b，即1字节）将 A0 的值加 1，使其指向下一个内存单元。这非常适合遍历数组/缓冲区。
问：为什么 add.b 0x30, D0 可以将数字变成 ASCII 字符？
答：在 ASCII 编码中，字符 '0' 到 '9' 的十六进制值分别是 0x30 到 0x39。因此，将数值 3 加上 0x30 就得到了 0x33，即字符 '3' 的 ASCII 码。
问：cmp.l 0x40, D4 和 cmp.l D4, 0x40 有什么区别？比较指令是如何影响标志位的？
答：M68k 的 cmp 指令执行的是 目标操作数 - 源操作数 (Destination - Source)。cmp.l 0x40, D4 执行的是 D4 - 0x40。如果 D4 大于等于 0x40，减法结果非负，且无借位，会触发 bge (大于等于) 分支。源和目标写反会导致条件判断逻辑完全相反。
3. 内存与溢出处理 (Память и обработка переполнения)
问：溢出错误码 0xCCCCCCCC 是随便选的吗？
答：不是。0xCC 是微软 Visual C++ 编译器在 Debug 模式下用来标记未初始化的堆栈内存的典型魔数。这里用它作为明显的错误标志，方便在调试时识别。
问：如果在 flush_run 中，缓冲区写了 63 和 64 字节时，溢出判断 cmp.l 0x40, D3 和 bge overflow 会怎么执行？
答：每次 flush 会增加 2 字节 (add.l 2, D3)。如果之前 D3 是 62，加 2 后变为 64 (0x40)。cmp.l 0x40, D3 计算结果为 0，Z 标志位置 1。bge (大于等于，N=V 时触发) 在结果为 0 时也会触发，因此会正确跳转到 overflow，防止缓冲区越界写入。
4. 代码优化与改进 (Оптимизация и улучшение кода)
问：这段代码有什么可以优化的地方？
答：
flush_run 和 flush_and_new 有大量重复代码（写入缓冲区的逻辑），可以提取为一个公共子程序，使用 jsr 调用以减少代码冗余。
如果输入流中本身就不包含数字字符 (0-9)，可以优化 RLE 格式，例如遇到单字符直接输出原字符，遇到连续字符才加数字前缀，这样可以避免 1A1B1C 这种膨胀现象。
输出缓冲区没有充分利用，如果采用双缓冲或流式处理，边压缩边输出，可以省略 output_buf，节省内存。
