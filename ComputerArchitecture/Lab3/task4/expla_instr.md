### 解释

Использование цикла для поэлементной проверки и сложения (как в Python) неэффективно для ассемблера по двум главным причинам:  
在汇编语言中，使用循环逐个检查元素并进行加法运算（如 Python）效率低下，主要有两个原因：

Скорость и сложность (O(n) против O(1)): Цикл требует выполнения множества инструкций (переход, вычисление остатка, ветвление, сложение) для каждого числа. При большом n (например, 90000) процессор выполнит 90000 итераций, что занимает много тактов. Математическая формула m*(m+1) вычисляет результат за фиксированное время (всего 3-4 инструкции), то есть работает за O(1) и выполняется мгновенно при любом n.  
速度和复杂度（O(n) vs. O(1)）：循环需要对每个数字执行多条指令（跳转、取余、分支、加法）。对于较大的 n 值（例如 90,000），处理器需要执行 90,000 次迭代，这将耗费大量时钟周期。而数学公式 m*(m+1) 只需固定时间（仅需 3-4 条指令）即可计算结果，这意味着它的运行时间为 O(1)，对于任何 n 值都是瞬时的。  
Сложность проверки переполнения: Если складывать числа по одному в цикле, нам пришлось бы проверять флаг переполнения после каждого сложения, что загромождает код и замедляет работу. При использовании формулы мы можем надежно проверить переполнение всего один раз — после финального умножения с помощью инструкций mul и mulh.
Поэтому замена цикла математической формулой — это стандартный метод оптимизации алгоритмов на уровне ассемблера.  
溢出检查复杂度：如果我们在循环中逐个加法，则每次加法后都必须检查溢出标志，这会使代码变得冗长且执行速度变慢。使用公式，我们只需在最后一次乘法运算后使用 mul 和 mulh 指令可靠地检查一次溢出即可。因此，用数学公式代替循环是程序集级别算法优化的标准方法。  


1. main 函数（主程序入口）
- 用处：程序的起点和总调度，负责初始化环境、读取输入、校验输入、调用核心逻辑、输出结果和停机。

- main (主体逻辑)：
初始化栈指针 sp = 0x1000 并对齐。
从 MMIO 地址 0x80 读取输入参数 n。
校验 n：如果 n ≤ 0，跳转到 main__invalid。
如果 n > 0，在栈上保存返回地址 ra，调用 sum_even 函数。
sum_even 返回后，恢复 ra，释放栈空间，并跳转到 main__write_output。
main__invalid (子标签：非法输入处理)：
当 n ≤ 0 时执行。
将返回值寄存器 a0 设为 -1，表示输入不合法。
main__write_output (子标签：统一输出口)：
无论是正常计算返回，还是非法输入返回，最终都会汇聚到此标签。
将 a0 中的结果（计算结果、-1 或 0xCCCCCCCC）写入 MMIO 输出端口 0x84。
执行 halt 永久停机。
2. sum_even 函数（核心计算逻辑）
- 用处：计算 1 到 n 的偶数和。采用数学公式 S = m * (m+1)（其中 m = n / 2）代替循环累加，并包含嵌套调用演示和完整的溢出检测机制。

sum_even (主体逻辑)：
建栈帧：分配 12 字节栈空间，保存 ra、s0、s1（被调用者保存寄存器）。
保存参数：将 n 从 a0 移动到 s0 防止被覆盖。
嵌套调用：保存当前 ra，调用 is_even(n) 演示嵌套调用，返回后恢复 ra（未使用 is_even 的返回值）。
公式计算：计算 m = n/2，计算 m * (m+1)，将低 32 位结果暂存于 t1。
溢出判断：提取乘积高 32 位（mulh），若高位非 0 或低位结果符号为负（正正得负），则跳转至 sum_even__overflow。
sum_even__ok (子标签：正常结果路径)：
当没有溢出时执行。
将 t1 中的正确结果移动到返回值寄存器 a0。
无条件跳转到 sum_even__return。
sum_even__overflow (子标签：溢出处理路径)：
当结果超出 32 位有符号整数范围时执行。
利用位运算技巧（0xFFFFFFFF XOR 0x33333333）构造出 0xCCCCCCCC 并存入 a0 作为溢出标记。
sum_even__return (子标签：函数统一返回出口)：
正常和溢出路径的最终汇聚点。
恢复 s1、s0、ra 的值。
释放 12 字节栈帧。
通过 jr ra 返回 main 函数。
3. is_even 函数（奇偶校验逻辑）
- 用处：判断传入的参数是否为偶数。这是一个为了满足“嵌套调用”要求而存在的辅助函数，虽然核心公式不需要它，但很好地演示了栈在多层函数调用中的保护作用。

is_even (主体逻辑)：
计算 n % 2（使用 rem 取余指令）。
如果余数等于 0，跳转到 is_even__yes。
如果不等于 0（即奇数），将返回值 a0 设为 0，并直接返回。
is_even__yes (子标签：偶数路径)：
当 n 是偶数时执行。
将返回值 a0 设为 1。
返回调用者（sum_even）。


一、 关于寄存器调用约定（极大概率被问）
1. 问：为什么你要把参数 n 从 a0 移动到 s0？直接用 a0 或者临时寄存器 t0 不行吗？

回答思路：不行。因为接下来我调用了 is_even 函数。根据 RISC-IV（及 RISC-V）的调用约定，a0 是参数和返回值寄存器，t0 是临时寄存器，它们都是 Caller-saved（调用者保存） 寄存器。is_even 函数内部很可能会修改 a0 和 t0 来计算并返回结果。如果我不把 n 转存到 s0（Callee-saved 被调用者保存寄存器），调用 is_even 后 n 的值就丢失了。虽然我在这道题中没有真正使用 is_even 的返回值，但为了遵循严格的调用约定和保护现场逻辑，必须使用 s0。
2. 问：你在 sum_even 函数开头保存了 ra、s0、s1，为什么必须保存它们？

回答思路：保存 ra 是因为 sum_even 内部使用了 jal 指令去调用 is_even，这会覆盖掉 main 返回时需要的 ra，所以必须压栈保存。保存 s0 和 s1 是因为它们是 Callee-saved 寄存器，规范要求被调用函数必须保证这些寄存器在返回时值与进入时一致。我既然在函数中使用了它们，就必须在出入口进行保存和恢复。
二、 关于嵌套调用与程序逻辑
3. 问：你调用了 is_even(n)，但并没有使用它的返回值来判断是否加到总和中，这不是多余的吗？为什么要写这段代码？

回答思路：从纯算法角度看确实是多余的，因为我后面直接用了数学公式 m*(m+1) 代替了循环累加。但是，实验要求演示嵌套调用和多层栈操作。保留这个调用是为了展示在嵌套调用时，内层函数会破坏外层函数的 ra，因此必须再次进行 ra 的压栈保护，这是理解函数调用栈帧的核心知识点。

三、 关于溢出检测（重点难点）
4. 问：你的溢出检测有两步（mulh 判断和 t1 < 0 判断），只做一步行不行？比如只判断 mulh 的结果是否为 0？

回答思路：不行。只判断 mulh 不够全面。mulh 保存的是乘积的高 32 位，如果高 32 位不为 0，说明结果超过了 32 位无符号数的范围（即>2^32−1），这肯定是溢出了。但是，如果乘积结果落在2^31到2^32−1
 之间，此时高 32 位（mulh）仍然是 0，但低 32 位（t1）的最高位（符号位）变成了 1。在 32 位有符号整数看来，这是一个负数。因为 m和m+1都是正数，乘积不可能是负数，出现正数相乘得负数就是典型的有符号溢出，所以必须加上 bgt t3, t1 这一步判断。

四、 关于指令集特性与常数构造
5. 问：溢出时返回 0xCCCCCCCC，为什么不直接用 lui 和 addi 拼 0xCCCCCCCC，而要用 0xFFFFFFFF XOR 0x33333333 这种奇怪的写法？

回答思路：这是 RISC-IV（RISC-V）指令集 12 位有符号立即数的限制。如果用 lui t0, 0xCCCCC，高 20 位没问题；但在执行 addi t0, t0, 0xCCC 时，0xCCC 的十进制是 3276，超出了 12 位有符号立即数的最大值 2047。CPU 硬件会将其视为负数进行符号扩展，导致加法计算结果完全错误。而 0x333 的十进制是 819，在 2047 范围内，不会有符号扩展问题。利用 0xFFFFFFFF XOR 0x33333333 = 0xCCCCCCCC 完美避开了这个陷阱。

6. 问：0(t0) 这种语法是什么意思？

回答思路：这是典型的基址+偏移量寻址方式。0(t0) 表示内存地址等于 基址寄存器 t0 的值 加上 偏移量 0。如果写成 4(t0)，则表示地址为 t0 + 4。lw 和 sw 指令只能使用这种寻址方式访问内存。
五、 边界与异常情况
7. 问：如果输入 
n=0
 或者负数，程序是怎么处理的？

回答思路：在 main 函数中，读取输入后立刻使用了 ble a0, zero, main__invalid 指令。如果 
n≤0
，程序会直接跳转到 main__invalid 标签，将 a0 置为 -1，并跳过所有计算直接输出，符合题目要求的“非法输入返回 -1”。
8. 问：你在 main 函数开头做了 addi sp, sp, -4 预留对齐，这有必要吗？

回答思路：这是良好的编程习惯（也符合许多 ABI 规范）。RISC-IV 的字长是 4 字节，lw 和 sw 要求地址必须是 4 字节对齐的（即地址最后两位为 00）。初始化 sp = 0x1000 是对齐的，减去 4 变成 0xFFC 仍然是对齐的。虽然在这个简单程序中不减 4 也能运行，但在复杂的嵌套调用中，保持栈指针始终 4 字节对齐是避免内存访问异常（Misaligned Access）的基础。
