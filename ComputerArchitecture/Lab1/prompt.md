Slide 2: Введение и проблема (引言与问题背景)
Содержимое слайда:
Проблема: Ошибки памяти — главная причина уязвимостей безопасности (около 70%).
Типы ошибок:
Пространственные (Spatial): Buffer overflow, Out-of-bounds.
Временные (Temporal): Use-after-free (UAF), Use-after-return (UAR).
Существующие решения: ASan, MSan (программные, высокие накладные расходы).
Новое решение: MTE (аппаратная поддержка).
Текст выступления:
Для начала определим проблему. Как мы знаем из курса системного программирования, более 70% критических уязвимостей безопасности связаны с ошибками работы с памятью. Они делятся на два основных типа: пространственные (spatial), такие как переполнение буфера (buffer overflow), и временные (temporal), такие как использование памяти после освобождения (use-after-free). До сих пор мы в основном использовали программные sanitizers, такие как AddressSanitizer (ASan). Они хорошо работают, но имеют огромные накладные расходы. MTE предлагает решить эту проблему на аппаратном уровне.

(首先定义问题。正如我们在系统编程课上学到的，超过70%的严重安全漏洞与内存错误有关。它们主要分为两类：空间错误（如缓冲区溢出）和时间错误（如释放后使用）。到目前为止，我们主要使用软件Sanitizer（如ASan）。它们虽然好用，但开销巨大。MTE提出在硬件层面上解决这个问题。)

Slide 3: Основные принципы MTE: Lock-and-Key (MTE基本原理：锁与钥匙)
Содержимое слайда:
Механизм "Замок и Ключ":
Ключ (Key): 4-битный тег в верхнем байте указателя (Logical Tag).
Замок (Lock): 4-битный тег в памяти (Allocation Tag).
Top Byte Ignore (TBI): Аппаратное игнорирование старшего байта указателя при вычислении адреса.
Гранулярность: 16 байт памяти = 1 тег.
Текст выступления:
Как это работает? MTE реализует механизм "замок и ключ" (lock-and-key). Память делится на блоки по 16 байт. Каждые 16 байт получают 4-битный "замок", который называется Allocation Tag. С другой стороны, указатель получает "ключ" — Logical Tag. Этот ключ хранится в верхнем байте 64-битного указателя. Фича Top Byte Ignore (TBI), которая уже была в ARM, позволяет процессору игнорировать этот верхний байт при расчете физического адреса, но использовать его для проверки безопасности. При каждой операции чтения или записи процессор аппаратно сравнивает ключ указателя и замок памяти.

(它是如何工作的？MTE实现了“锁与钥匙”机制。内存被划分为16字节的块。每16字节获得一个4位的“锁”，称为分配标签。另一方面，指针获得一个“钥匙”——逻辑标签。这个钥匙存储在64位指针的最高字节。TBI（最高字节忽略）特性允许处理器在计算物理地址时忽略这个最高字节，但用它来进行安全检查。在每次读写操作时，处理器会在硬件层面比较指针的钥匙和内存的锁。)

Slide 4: Режимы работы MTE (MTE的工作模式)
Содержимое слайда:
SYNC (Синхронный): Исключение возникает сразу при несовпадении тегов. (Высокая точность).
ASYNC (Асинхронный): Ошибка записывается в регистр, исключение откладывается. (Низкий overhead).
ASYMM (Асимметричный): Чтение (Load) — синхронно, Запись (Store) — асинхронно. (Золотая середина).
Текст выступления:
Поскольку проверка тегов требует дополнительных ресурсов, Arm внедрила три режима работы. Первый — SYNC (синхронный). При несовпадении тегов процессор мгновенно генерирует исключение. Идеально для отладки, но дает средний overhead. Второй — ASYNC (асинхронный). Процессор просто записывает факт ошибки в специальный регистр и продолжает работу. Это дает минимальный overhead, почти незаметный для программы. Третий — ASYMM (асимметричный). Чтение проверяется синхронно, а запись — асинхронно. Это так называемая "золотая середина", которая используется, например, в Android по умолчанию.

(由于标签检查需要额外资源，Arm引入了三种工作模式。第一是同步模式，标签不匹配时处理器立即产生异常。非常适合调试，但有一定开销。第二是异步模式，处理器只将错误记录在特殊寄存器中并继续运行。开销极小。第三是非对称模式，读取同步检查，写入异步检查。这是所谓的“折中方案”，例如Android默认就使用这种模式。)

Slide 5: Обнаружение пространственных ошибок (空间错误检测)
Содержимое слайда:
Пример: Buffer Overflow
Механизм:
Аллокатор выделяет буфер → Tag A.
Соседние 16 байт → Tag B.
Доступ за пределами буфера: Key A 

= Tag B → ОШИБКА (Fault).
Текст выступления:
Посмотрим, как MTE ловит пространственные ошибки на примере Buffer Overflow. Когда аллокатор выделяет буфер, он присваивает ему, например, Тег А. Соседние 16 байта в памяти, которые не принадлежат нашему буферу, получают Тег Б. Если программа пытается прочитать данные за пределами буфера, аппаратура видит, что Ключ А не совпадает с Замком Б, и вызывает fault. Баг пойман.

(看看MTE如何以缓冲区溢出为例捕捉空间错误。当分配器分配缓冲区时，它赋予其标签A。内存中不属于我们缓冲区的相邻16字节获得标签B。如果程序试图读取缓冲区之外的数据，硬件会发现钥匙A与锁B不匹配，并触发错误。Bug被捕获了。)

Slide 6: Обнаружение временных ошибок (时间错误检测)
Содержимое слайда:
Пример: Use-After-Free (UAF)
Механизм:
Память выделена: Key A = Tag A.
Вызов free(): Аллокатор меняет память на Tag B.
Доступ по старому указателю: Key A 

= Tag B → ОШИБКА (Fault).
Текст выступления:
Теперь временные ошибки — Use-After-Free. Это самый сложный тип багов. Память выделяется с Тегом А. Указатель имеет Ключ А. Когда мы вызываем free(), аллокатор не очищает память физически (это долго), он просто меняет Allocation Tag в памяти на Тег Б. Если программа позже пытается использовать старый указатель (Ключ А), происходит несовпадение с Тегом Б, и процессор прерывает выполнение. Баг обнаружен аппаратно!

(现在是时间错误——释放后使用。这是最难的Bug类型。内存分配时带有标签A，指针有钥匙A。当我们调用free时，分配器并没有物理清空内存（这很慢），它只是把内存中的分配标签改成了标签B。如果程序之后尝试使用旧指针（钥匙A），就会与标签B不匹配，处理器中断执行。Bug在硬件层面被发现了！)

Slide 7: Интеграция с минимальными правками кода (最小代码改动的集成)
Содержимое слайда:
Изменения в Аллокаторе: Установка тегов при malloc() и смена при free() (инструкции STG/STZG).
Изменения в ОС: Передача прав через системные вызовы (например, prctl в Linux).
Изменения в Приложении: 0 строк кода (достаточно перекомпиляции).
Текст выступления:
Важнейшее преимущество MTE — легкость внедрения. Вам не нужно переписывать исходный код приложения. Единственные изменения нужны в аллокаторе памяти: добавить установку тегов при выделении и их изменение при освобождении с помощью новых инструкций. Операционная система должна лишь разрешить использование тегов для процесса. Для самого разработчика приложения — это просто перекомпиляция с новым флагом компилятора.

(MTE最重要的优势是易于部署。你不需要重写应用程序的源代码。唯一的修改在于内存分配器：在分配时设置标签，释放时更改标签。操作系统只需允许进程使用标签即可。对于应用程序开发者来说，只需要用新的编译器标志重新编译。)

Slide 8: MTE vs ASan: Накладные расходы (MTE与ASan的性能开销对比)
Содержимое слайда:
Метрика	AddressSanitizer (ASan)	MTE (ASYNC)	MTE (SYNC)
CPU Overhead	+50% ... +200%	~2% ... 5%	~10% ... 15%
Memory Overhead	+2x (Теневая память)	~3% ... 5%	~3% ... 5%
Хранение тегов	Отдельная область памяти	В spare bits кэша L1/L2	В spare bits кэша L1/L2
Текст выступления:
Давайте сравним MTE с популярным ASan по данным из исследовательских статей. По CPU: ASan замедляет программу от 50% до 200% из-за инструментации каждого доступа к памяти. MTE в асинхронном режиме замедляет всего на 2-5%, так как проверка идет параллельно с вычислением адреса. По памяти: ASan требует удвоения памяти для так называемой "теневой памяти". MTE использует лишь 3-5% памяти, потому что 4-битные теги хранятся прямо в кэшах процессора (в неиспользуемых битах), вообще не занимая основную оперативную память!

(根据研究论文的数据，我们将MTE与流行的ASan进行对比。CPU方面：ASan由于对每次内存访问都进行插桩，会使程序变慢50%到200%。MTE在异步模式下仅变慢2-5%，因为检查是与地址计算并行进行的。内存方面：ASan需要将内存翻倍以用于所谓的“影子内存”。MTE仅使用3-5%的内存，因为4位标签直接存储在处理器缓存中（未使用的位中），根本不占用主存！)

Slide 9: Сравнение в реальных сценариях (基于实际研究场景的对比)
Содержимое слайда:
Точность: MTE (SYNC) атомарно ловит 100% ошибок на уровне доступа. ASan может пропустить UAF из-за race conditions.
Сложность внедрения: ASan требует меньше настроек стека/глобальных переменных. MTE требует аккуратной инициализации тегов во всех секциях памяти (иначе ложные срабатывания).
Источники: Arm MTE Performance in Practice (arXiv), Project Zero: MTE As Implemented.
Текст выступления:
Как отмечает команда Project Zero в своем анализе MTE, аппаратное решение надежнее программного. ASan может пропустить Use-After-Free из-за состояний гонки (race conditions), так как между проверкой и доступом к памяти может пройти время. MTE делает это атомарно на уровне процессора. Однако у MTE есть сложности: нужно аккуратно инициализировать теги для стека и глобальных переменных, иначе будут ложные срабатывания.

(正如Project Zero团队在分析中指出的，硬件解决方案比软件更可靠。ASan可能会因为竞态条件而漏掉UAF，因为在检查和访问内存之间可能会有时间差。MTE在处理器级别原子性地完成了这一操作。然而MTE也有难点：需要小心地初始化栈和全局变量的标签，否则会出现误报。)

Slide 10: Простые оптимизации MTE (MTE的简单优化)
Содержимое слайда:
Skipping Tag Checks: Инструкции LDG/STG и LDGM/STGM для безопасных массовых операций без проверки.
Поддержка компиляторов: Флаг -fsanitize=memtag в GCC/Clang (автоматическая расстановка инструкций).
Выравнивание: Аллокация памяти строго по границе 16 байт (alignas(16)).
Текст выступления:
Существуют простые оптимизации. Например, если мы уверены в безопасности памяти (например, копируем большой буфер внутри одной структуры), мы можем использовать инструкции без проверки тегов для ускорения. Современные компиляторы GCC и Clang с флагом -fsanitize=memtag сами понимают, где можно пропустить проверки, а где нужно вставить инструкции для установки тегов.

(存在一些简单的优化。例如，如果我们确信内存是安全的（比如在同一个结构体内复制大缓冲区），我们可以使用不带标签检查的指令来加速。现代编译器GCC和Clang加上-fsanitize=memtag标志后，自己能理解哪里可以跳过检查，哪里需要插入设置标签的指令。)

Slide 11: Поддержка в Android и Linux (Android与Linux中的支持)
Содержимое слайда:
Android (AOSP):
Начиная с Android 11/12, MTE включено по умолчанию для нативного кода.
Режим: ASYMM (баланс скорости и безопасности).
Linux:
Системный вызов: prctl(PR_SET_TAGGED_ADDR_CTRL...).
Выделение памяти: Флаг PROT_MTE в mmap().
Текст выступления:
MTE уже активно используется в промышленности. В Android (AOSP), начиная с 11-й версии, MTE включено по умолчанию в асимметричном режиме для всех нативных библиотек системы. Это защищает миллионы устройств без заметной потери батареи. В ядре Linux добавлена полная поддержка: через системный вызов prctl процесс может включить тегирование, а при вызове mmap с флагом PROT_MTE выделяется память с поддержкой тегов.

(MTE已在工业界积极使用。在Android中，从第11版开始，MTE默认以非对称模式为系统所有原生库启用。这在没有明显耗电增加的情况下保护了数百万设备。在Linux内核中添加了完整支持：通过prctl系统调用进程可以启用标记，而在使用带有PROT_MTE标志的mmap调用时，会分配支持标签的内存。)

Slide 12: Рекомендации по использованию (使用建议)
Содержимое слайда:
Тестирование (CI/CD):
Режим: SYNC или ASYMM.
Цель: Максимальная детекция багов, точное указание строки кода.
Продакшен (Production):
Режим: ASYNC.
Цель: Минимальный удар по производительности, сбор статистики о багах у реальных пользователей.
Совместное использование: MTE (в проде) + ASan (в тестах).
Текст выступления:
Как мы рекомендуем использовать MTE? В ходе тестирования и в CI/CD системах используйте синхронный режим (SYNC) — он сразу укажет на строку кода с ошибкой. В продуктивной среде (в продакшене) используйте асинхронный режим (ASYNC). Пользователи не заметят замедления, а вы будете собирать логи о реальных багах, происходящих на устройствах клиентов.

(我们如何建议使用MTE？在测试期间和CI/CD系统中，使用同步模式——它会立即指出错误的代码行。在生产环境中，使用异步模式。用户不会注意到变慢，而你可以收集发生在客户设备上的真实Bug的日志。)
