Ключевые свойства архитектуры

16

Tag Granule — 16 байт

Каждые 16 байт физической памяти получают один 4-битный тег. Выравнивание объектов до 16 байт гарантирует детерминированное обнаружение соседних переполнений.

4

4-битный тег — 16 возможных значений

Logical tag в старшем байте указателя через TBI — без расширения размера указателя и изменения ABI. Вероятность ложного совпадения: 1/16 ≈ 6.25%.

IRG

Аппаратная генерация случайных тегов

Инструкция IRG генерирует случайный тег в регистр. Аллокатор всегда назначает соседним блокам разные теги — последовательные переполнения обнаруживаются с вероятностью 1.

NEW

Normal Tagged Memory — новый тип памяти

В Armv8.5-A добавлен новый тип «Normal Tagged Memory». Только обращения к нему проверяются; существующий код в dynamically linked системах выигрывает без перекомпиляции.

03 · ISA Extensions

Новые инструкции Armv8.5-A для MTE

Три категории инструкций: работа с тегами, арифметика указателей, системный уровень

Работа с тегами памяти
IRG Генерация случайного тега в регистр — основа статистической гарантии MTE
STG / STZG Запись тега в память без модификации данных / с обнулением данных
LDG Чтение тега из памяти в регистр
ST2G / STZ2G Операции на двух гранулах — эффективнее для крупных блоков
STGP Совместная запись тега и данных в память одной инструкцией
GMI Управление excluded-множеством тегов для специальных значений
Арифметика указателей / стек
ADDG Сложение с раздельной арифметикой над адресом и тегом — создание tagged адресов на стеке
SUBG Вычитание с раздельной арифметикой — создание субадресов в пределах tagged-объекта
SUBP(S) 56-битное вычитание с игнорированием тега — корректная арифметика разниц указателей

Стратегия stack tagging

При входе в функцию: IRG → случайный тег. ADDG/SUBG → теги для каждого slot. Соседние объекты = разные теги → последовательный overflow всегда обнаружен.

Системный уровень (EL1+)
LDGM Массовое чтение тегов — только privileged. Для сериализации (swap to non-tag storage)
STGM Массовая запись тегов — инициализация, восстановление из swap
STZGM Zero + tag за одну операцию — высокоэффективная инициализация страниц в ядре ОС

Cache maintenance

MTE добавляет cache maintenance операции для работы с тегами на уровне cache-линий — эффективная batch-обработка.

05 · Spatial Safety

Пространственная
безопасность

Нарушение: объект доступен вне своих границ

Buffer Overflow на стеке

Перезапись return address → ROP / stack smashing. Основа эксплойтов с 1988 года (Worm Морриса).

Heap Buffer Overflow

Перезапись метаданных аллокатора или данных соседнего объекта — heap metadata corruption.

Как MTE обнаруживает

→Аллокатор назначает соседним блокам разные теги (гарантировано)
→Последовательный overflow всегда обнаруживается (детерминировано)
→«Дикий» произвольный доступ — вероятностно 15/16 ≈ 93.75%

05 · Temporal Safety

Временна́я
безопасность

Нарушение: ссылка используется после free()

Use-After-Free

Висячий указатель на освобождённый объект. Если память переназначена → перезапись нового объекта (vtable, function pointer → type confusion).

Use-After-Return

Указатель на объект стека, использованный после возврата из функции — стек-фрейм уже перезаписан.

Как MTE обнаруживает

→При free() аллокатор назначает памяти новый тег
→Висячий указатель хранит старый тег — несоответствие неизбежно
→Обращение через висячий указатель → немедленное исключение

06 · Deployment

Развёртывание MTE в программном обеспечении

Два уровня интеграции с разными требованиями к изменению кода

Heap Tagging

Без перекомпиляции

→
В dynamically linked системах — только ядро ОС и libc. Существующие бинари не трогаются. malloc / free / calloc / realloc
→
Linux kernel: поддержка Normal Tagged Memory в mmap, удаление тегов при syscall address space операциях, обработка SIGSEGV по аналогии с translation fault
→
Memory copy / string функции модифицированы для защиты от overread источника memcpy / strcpy / memmove
→
Android NDK API: выбор режима для текущего процесса android_mallopt(M_BIONIC_SET_HEAP_TAGGING_LEVEL)
Stack Tagging

Требует перекомпиляции

→
Компилятор вставляет IRG при входе в функцию для генерации случайного тега стек-фрейма
→
ADDG/SUBG создают tagged адреса для каждого stack slot с уникальным смещением тега — соседние объекты имеют гарантированно разные теги
→
Alignment объектов увеличивается до 16 байт — небольшое увеличение потребления стека (benchmarks: обычно незначительное)
→
Unchecked: SP-based immediate addressing — статически доказуемо корректно компилятором, проверка избыточна
→
Требуется BTI + PAC для защиты stack pointer от hijacking — MTE не защищает SP по архитектурному допущению
SYNC + collect → максимум информации при тестировании
ASYNC + terminate → баланс overhead / защита в продакшене
Crash → restart in SYNC → адаптивная диагностика

09 · Практика

Оптимизации и ограничения MTE

Рекомендации по снижению overhead и границы применимости технологии

✓ Рекомендуется

→

Записывать теги и данные вместе

Использовать STZGM для zero+tag за одну операцию при инициализации страниц ядром. Избегать раздельных операций записи данных и тегов.

→

Минимизировать de-allocation / re-allocation

Частые malloc/free усиливают overhead MTE: фиксированные затраты на аллокацию растут, существующие perf-проблемы amplify.

→

Избегать больших fixed-size стек-буферов

Буферы типа PATH_MAX (4096 байт) часто содержат короткие строки. Выравнивание до 16 байт при редком использовании → напрасная запись тегов.

→

Избегать over-allocation address space

Если ПО резервирует гигабайты address space, но пишет мало — теги всё равно пишутся. С MTE это дороже, чем без него.

✗ Ограничения

!

Вероятностная, не абсолютная защита

4 бита = 16 значений тегов. Вероятность случайного совпадения при «диком» доступе: 1/16. При N запусках вероятность → 100%.

1/16 false negative rate
!

Не защищает алгоритмические уязвимости

MTE работает с нарушениями memory safety, но не с логическими ошибками, инъекциями, или уязвимостями в коде без обращений к памяти.

!

Stack требует перекомпиляции + BTI + PAC

Архитектура предполагает trustworthy stack pointer. Без BTI и PAC атакующий может контролировать SP и обойти MTE для стека.

!

Только Arm — нет аналога на x86/RISC-V

MTE специфична для Armv8.5-A+. Intel MPX (deprecated 2019) и AMD не имеют функционального эквивалента в production silicon сегодня.
