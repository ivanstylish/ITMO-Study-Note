Структура слайдов + тезисы для выступления
Слайд 1 — Титульный
«Добрый день. Тема доклада — Arm Memory Tagging Extension, аппаратный механизм обнаружения ошибок памяти, появившийся в Armv8.5-A. Расскажу, почему это важно, как это работает и почему это реально используется в вашем смартфоне прямо сейчас.»
Слайд 2 — Масштаб проблемы
«70% уязвимостей Microsoft и 75%+ Android — это ошибки памяти в C/C++. Переписать полмиллиарда строк кода невозможно. Нужно аппаратное решение. Все эти уязвимости делятся на два класса: spatial (выход за границы) и temporal (обращение после free).»
Слайд 3 — Механизм Lock & Key
«MTE работает как замок и ключ. На каждые 16 байт физической памяти хранится 4-битный тег-замок. Указатель несёт 4-битный тег-ключ в верхнем байте — это называется Top Byte Ignore. При каждом load/store процессор сравнивает их. Несовпадение — SIGSEGV.»
Слайд 4 — Новые инструкции
«Для реализации MTE в ISA добавлены три группы инструкций: работа с тегами памяти (IRG, STG, LDG), арифметика для стека (ADDG/SUBG), и системный уровень (STGM — только EL1+, для ядра ОС).»
Слайд 5 — Три режима
«SYNC — точное исключение на нарушающей инструкции, overhead 5–25%, для разработки. ASYNC — несоответствие копится в регистре, проверяется при syscall, overhead ~1–2%, пригоден для продакшена. ASYMM — гибрид: sync на чтениях, async на записях.»
Слайд 6 — Spatial vs Temporal
«Spatial: аллокатор гарантирует разные теги у соседних блоков → последовательный overflow всегда обнаруживается. Temporal: при free() тег блока меняется → висячий указатель несёт старый тег → use-after-free обнаружен.»
Слайд 7 — Развёртывание
«Heap tagging не требует перекомпиляции в динамически слинкованных системах — только ядро и libc. Stack tagging требует перекомпиляции, а также BTI и PAC для защиты stack pointer.»
Слайд 8 — MTE vs ASan
«ASan замедляет в ~2 раза и удваивает RAM. MTE ASYNC — менее 2% CPU overhead и менее 3% RAM, пригоден для продакшена. Данные из arXiv:2601.11786 (2026) на Pixel 8/9 и AmpereOne.»
Слайд 9 — Android & Linux
«Pixel 8 (2023) — первый массовый телефон с MTE в железе. Android NDK: android_mallopt() для выбора режима. Linux: prctl(PR_SET_TAGGED_ADDR_CTRL). Apple A19 (iPhone 17, 2025) тоже поддерживает MTE.»
Слайд 10 — Оптимизации и ограничения
«Ключевые оптимизации: писать теги и данные вместе (STZGM), избегать избыточных malloc/free, не выделять огромные буферы на стеке. Ограничения: 4 бита = 1/16 вероятность false negative при "диком" доступе, не защищает алгоритмические уязвимости, только Arm.»
Слайд 11 — Итоги
«MTE — это аппаратный Lock & Key, работающий с минимальным overhead, без изменения кода для heap, с поддержкой в Linux, Android и iOS экосистемах. Это не панацея, но первый реально продакшен-готовый инструмент детекции ошибок памяти на уровне железа.»

Вопросы аудитории и ответы
В: Почему 4 бита, а не больше? Ведь 1/16 — это мало?
О: Это компромисс: 4 бита занимают ровно один ниббл старшего байта указателя через TBI, не расширяя ABI. При последовательных аллокациях (соседние блоки всегда имеют разные теги) детекция гарантирована. 1/16 — только для «диких» произвольных обращений, но при многократных запусках тест-сьюта вероятность обнаружить стремится к 100%.
В: Чем MTE отличается от AddressSanitizer (ASan)?
О: ASan — программная инструментация компилятора, shadow memory, 2× overhead CPU, +50-100% RAM, непригодна для продакшена. MTE — аппаратная проверка при каждом load/store, <2% ASYNC overhead, <3% RAM, работает в продакшене без перекомпиляции для heap.
В: Почему stack tagging требует перекомпиляции, а heap — нет?
О: Heap: malloc/free — это библиотечные функции, которые просто заменяются. Stack: адреса объектов на стеке вычисляются компилятором через ADDG/SUBG, нужна статическая информация о layout стека — только компилятор это знает.
В: Что произойдёт, если тег памяти случайно совпадёт с тегом "не того" указателя?
О: False negative — уязвимость не обнаружится при данном запуске. Вероятность 1/16 ≈ 6.25%. Аллокатор может использовать pseudo-random seed, чтобы при повторных запусках использовались разные теги, и за N запусков вероятность обнаружить ошибку → 100%.
В: Как MTE взаимодействует с виртуальной памятью и swap?
О: Теги хранятся в физической памяти (дополнительные 4 бита на 16 байт) и когерентны через AMBA CHI. При swap ядро использует LDGM/STGM для сериализации тегов совместно с данными — это одна из причин, зачем нужны системные инструкции.
В: Почему SYNC может давать overhead до 25%?
О: В SYNC режиме load превращается в: load-tag → check-tag → load-value, причём порядок строгий — нельзя начать нагрузку данных до проверки. Это создаёт сериализацию в конвейере. arXiv:2601.11786 показал до 6.64× на store-heavy microbenchmarks из-за именно этой архитектурной особенности текущих реализаций.
