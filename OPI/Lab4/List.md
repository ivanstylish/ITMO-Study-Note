## 🔑 Ответы на вопросы к защите лабораторной работы

### 1. Мониторинг и профилирование. Основные понятия. Отличия мониторинга от профилирования.

**Мониторинг** — это **непрерывное** наблюдение за работой системы (ОС, JVM, приложения) с целью сбора метрик производительности. Мониторинг показывает *что* происходит — загрузка CPU, память, I/O.

**Профилирование** — это **целенаправленный** анализ поведения конкретного приложения: какие методы сколько времени занимают, сколько объектов создаётся, какие потоки конкурируют за блокировки. Профилирование показывает *почему* происходит.

| Критерий | Мониторинг | Профилирование |
|---|---|---|
| Цель | Наблюдение за системой в целом | Анализ узких мест в конкретном приложении |
| Интрузивность | Обычно неинтрузивен | Может быть интрузивным |
| Уровень | ОС, JVM, сеть, диск | Методы, объекты, потоки приложения |
| Примеры инструментов | top, vmstat, JConsole | VisualVM, NetBeans Profiler, jmap |

**Пример:** Мониторинг показал, что CPU загружен на 100% — это мониторинг. Потом запустили профилировщик и обнаружили, что 17% CPU жрёт `ArrayList.contains()` — это профилирование.

### 2. Инфраструктура для организации мониторинга и профилирования в составе JDK. JMX.

**JMX (Java Management Extensions)** — стандартная технология Java для управления и мониторинга приложений. Является частью JDK начиная с Java 5.

**Ключевые компоненты JMX:**

- **MBeans (Managed Beans)** — Java-объекты, которые представляют управляемые ресурсы (см. вопрос 3)
- **MBean Server** — регистр (контейнер) для всех MBean-объектов в JVM
- **JMX Agent** — сервер, к которому подключаются внешние инструменты (JConsole, VisualVM)
- **Connectors** — механизм удалённого подключения к JMX Agent (RMI, JMXMP)

**Инструменты JDK для мониторинга:**
- `jps` — список запущенных Java-процессов
- `jstat` — статистика JVM (GC, компиляция, загрузка классов)
- `jmap` — дамп кучи (heap dump)
- `jhat` — анализ heap dump
- `jstack` — дамп стеков потоков
- `JConsole` — GUI-инструмент мониторинга через JMX
- `VisualVM` — расширенный GUI-профилировщик

**Для профилирования через JDK:**
- `-verbose:gc -XX:+PrintGCDetails` — лог сборщика мусора
- `-XX:+PrintCompilation` — лог JIT-компиляции
- `-verbose:class` — лог загрузки классов

### 3. MBeans. Основные понятия. Архитектура фреймворка.

**MBean (Managed Bean)** — специальный Java-объект, следующий определённым соглашениям, который предоставляет атрибуты, операции и уведомления для мониторинга и управления.

**Виды MBeans:**
- **Standard MBean** — самый простой вид; нужно создать интерфейс `XMBean` и класс `X`
- **Dynamic MBean** — реализует интерфейс `DynamicMBean`; структура определяется во время выполнения
- **Model MBean** — конфигурируется через дескрипторы
- **Open MBean** — использует только "открытые" типы данных
- **MXBean** — разновидность, использующая стандартные типы Java (используется в `java.lang.management`)

**Архитектура JMX (три уровня):**

```
┌─────────────────────────────────────────┐
│  Уровень управления (Managers)          │  ← JConsole, VisualVM, внешние системы
│  (подключаются через Connectors/RMI)    │
├─────────────────────────────────────────┤
│  Уровень агента (Agent Layer)           │  ← MBean Server + JMX Agent
│  MBeanServer = реестр всех MBeans       │
├─────────────────────────────────────────┤
│  Уровень ресурсов (Resource Layer)      │  ← MBeans (Standard, Dynamic, MXBean)
│  Описывают управляемые ресурсы JVM/приложения │
└─────────────────────────────────────────┘
```

**Встроенные MXBeans в JDK** (пакет `java.lang.management`):
- `MemoryMXBean` — состояние памяти heap/non-heap
- `ThreadMXBean` — потоки, deadlock detection
- `GarbageCollectorMXBean` — статистика GC
- `RuntimeMXBean` — информация о JVM
- `OperatingSystemMXBean` — CPU, OS

**Пример получения данных через MXBean:**
```java
MemoryMXBean memBean = ManagementFactory.getMemoryMXBean();
long heapUsed = memBean.getHeapMemoryUsage().getUsed(); // байты
```

### 4. Утилита JConsole. Возможности, область применения.

**JConsole** — графический инструмент мониторинга JVM, входит в состав JDK (`$JAVA_HOME/bin/jconsole`). Подключается к запущенной JVM через JMX.

**Возможности JConsole:**
- Мониторинг **памяти** (heap, non-heap, PermGen/Metaspace по поколениям)
- Мониторинг **потоков** (количество, имена, deadlock detection)
- Мониторинг **классов** (загружено/выгружено)
- Мониторинг **CPU** приложения
- Просмотр и управление **MBeans** (вызов операций, изменение атрибутов)
- Принудительный вызов **GC**
- **Подключение к удалённым JVM** (через RMI/JMX)

**Область применения:**
- Быстрый первичный осмотр состояния JVM
- Поиск утечек памяти (наблюдать за ростом heap)
- Проверка наличия deadlock'ов
- Управление приложением через MBeans в реальном времени

**Как запустить с поддержкой JMX:**
```bash
java -Dcom.sun.management.jmxremote \
     -Dcom.sun.management.jmxremote.port=9999 \
     -Dcom.sun.management.jmxremote.ssl=false \
     -Dcom.sun.management.jmxremote.authenticate=false \
     -jar MyApp.jar
```

### 5. Утилита Visual VM. Возможности, область применения.

**VisualVM** — расширенный визуальный инструмент для анализа Java-приложений. Начиная с JDK 9 распространяется отдельно (https://visualvm.github.io).

**Возможности VisualVM:**
- Всё, что умеет JConsole +
- **CPU профилирование** — какие методы сколько времени занимают (с call tree)
- **Memory профилирование** — сколько объектов каждого класса создано, занятая память
- **Heap dump** — снимок кучи с анализом объектов
- **Thread dump** — снимок состояния всех потоков
- **VerboseGC** плагин — визуализация работы сборщика мусора в реальном времени
- Поддержка **плагинов** (VisualGC, BTrace, MBeans, и др.)
- Профилирование **удалённых** JVM через JMX/jstatd

**Область применения:**
- Углублённое профилирование (поиск "горячих" методов)
- Поиск утечек памяти (heap dump → поиск задержанных объектов)
- Анализ работы GC
- Профилирование на удалённых серверах

**Отличие от JConsole:** VisualVM — профилировщик, JConsole — мониторинг. VisualVM интрузивен (замедляет приложение), JConsole — нет.

### 6. Удалённый мониторинг и профилирование приложений на платформе Java.

**Способы удалённого подключения:**

**1. Через JMX/RMI (стандартный способ)**
```bash
# Запуск приложения с удалённым JMX
java -Dcom.sun.management.jmxremote \
     -Dcom.sun.management.jmxremote.port=9999 \
     -Dcom.sun.management.jmxremote.ssl=false \
     -Dcom.sun.management.jmxremote.authenticate=false \
     MyApp
```
Затем JConsole или VisualVM подключаются по адресу `host:9999`

**2. Через jstatd (для VisualVM)**
```bash
# На сервере запустить jstatd
jstatd -J-Djava.security.policy=jstatd.all.policy
```
VisualVM видит все JVM на удалённом хосте.

**3. Через Java Attach API**
Позволяет подключаться к уже запущенной JVM без предварительной конфигурации (если JVM запущена на той же машине или с нужными правами).

**Инструменты для удалённого мониторинга:**
- **JConsole** — поддерживает JMX Remote из коробки
- **VisualVM** — поддерживает JMX Remote + jstatd
- **jstat** — `jstat -gcutil <pid>@<host>:<port> 1000` (через jstatd)

**Безопасность:**  
В продакшене обязательно включать SSL и аутентификацию:
```bash
-Dcom.sun.management.jmxremote.ssl=true
-Dcom.sun.management.jmxremote.authenticate=true
-Dcom.sun.management.jmxremote.password.file=/path/jmxremote.password
```

## 1. От чего зависит скорость выполнения программ?

**Три основных аспекта:**

**1. Системный и архитектурный аспект:**
- Архитектура (монолитная, n-tier, потоковая)
- Виртуализация и кластеризация
- *Пример:* В трёхзвенной архитектуре медленный слой БД замедляет всю бизнес-логику

**2. Низкоуровневый аппаратный аспект:**
- Частота, количество процессоров/ядер
- Объём и скорость кэшей, памяти, диска
- *Пример:* При 2% промахов кэша производительность падает ниже 60% от идеальной!

**3. Программный аспект:**
- Выбор алгоритмов и структур данных (пузырьковая сортировка vs Qsort)
- Пулы, многопоточность, блокировки

**4. Человеческий фактор:**
- Ошибки разработчиков, неправильные архитектурные решения

---

## 2. Влияние средств наблюдения на результаты

Анализ производительности — это **научный эксперимент** со строгим алгоритмом:

1. Выбрать критерии оценки (что измеряем)
2. Выбрать средства измерения
3. Выбрать/создать нагрузку
4. Провести анализ результатов
5. Изменить **один** параметр или переписать **один** участок кода
6. Повторить до удовлетворения

**Виды средств измерения по интрузивности:**

| Вид | Описание | Пример |
|---|---|---|
| **Неинтрузивные** | Не влияют на результат | Счётчики ОС |
| **Слабоинтрузивные** | Незначительно искажают | Вольтметр в электросети |
| **Интрузивные** | Существенно меняют поведение | Весы у нечестного продавца |

> ⚠️ Важно: практически любое измерение производительности **влияет** на результат. Нагрузку надо делать максимально эквивалентной реальной.

## 3. Основные характеристики и ключевые понятия

| Термин | Определение | Пример |
|---|---|---|
| **Время отклика (latency)** | От запроса до первых результатов | Время от вставки карты до выдачи первой купюры |
| **Время полного обслуживания** | От запроса до получения всего результата | До получения всей суммы в банкомате |
| **Пропускная способность** | Макс. запросов в единицу времени | 1000 req/s через API |
| **Утилизация (%util, %busy)** | Доля времени ресурс занят работой | Банкомат 9 мин из 10 = 90% |
| **Ожидание (%wait)** | Среднее время очереди ≠ 0 | Очередь к банкомату |
| **Точка насыщения** | Макс. производительность системы | После неё — резкое падение |
| **Масштабируемость** | Насколько можно нарастить нагрузку | |
| **Эффективность** | Полезная работа / вся работа | CPU 100% ≠ эффективная работа |
| **Ускорение (speedup)** | Во сколько раз быстрее после изменений | |
| **Memory footprint (Java)** | Стратегии использования памяти | Неправильная работа с памятью = резкое падение скорости |

## 4. Классический "нисходящий" метод поиска узких мест

Система исследуется от общего к частному (top-down):

```
1. Исключение ошибок аппаратуры и администратора
   └─ Системные журналы, конфиги
        (Пример: дефект кабеля → повторное чтение одного участка диска)

2. Общесистемный и межузловой мониторинг
   └─ CPU, память, I/O, сеть, ОС, виртуализация

3. Мониторинг приложения
   └─ Алгоритмы, API, многопоточность, блокировки, синхронизация

4. Мониторинг микроархитектуры (если предыдущего мало)
   └─ Выравнивание данных, оптимизация кэшей, конвейер
      (нужно знать ассемблер и архитектуру процессора)
```

## 5. Пирамида памяти (иерархия)

Сверху — быстрая, дорогая, маленькая. Снизу — медленная, дешёвая, большая.

| Уровень | Объём | Время доступа | Управление |
|---|---|---|---|
| CPU регистры | 100-1000 б | < 1 нс | Компилятор |
| L1 Cache | 32-128 КБ | 1-4 нс | Аппаратура |
| L2-L3 Cache | 0.5-32 МБ | 8-20 нс | Аппаратура |
| Основная память (RAM) | 0.5 ГБ - 4 ТБ | 60-200 нс | malloc/free |
| SSD | 128 ГБ - 1 ТБ | 25-250 мкс | open/read/write |
| Жёсткий диск | 0.5-4 ТБ | 5-20 мс | open/read/write |
| Магнитные ленты | 1-6 ТБ/к | 1-240 с | Программно |

> 🔑 **Главный вывод:** При 2% cache miss производительность падает ниже 60% от идеальной!

**Аналогия:** Если такт процессора = 1 секунда диалога, то:
- L1 кэш = 4 секунды (нормально)
- L2 кэш = 30 секунд (странно)
- RAM = 3-10 минут (неприемлемо)
- SSD = 5 дней
- HDD = 4 месяца

**Вывод:** Минимизировать обращения к нижним уровням пирамиды — один из главных способов ускорить программу.

## 6. Системный мониторинг производительности

Системный мониторинг обычно **неинтрузивен** (счётчики собираются ОС в любом случае).

**Что мониторится:**

| Подсистема | Параметры |
|---|---|
| **CPU** | user%, system%, idle%, ctxt_sw, interrupts, load average |
| **I/O** | bytes read/write/s, operations/s, wait time |
| **Виртуальная память** | free, buff, cache, pagein/s, pageout/s, scan time |
| **Сеть** | TX/RX потоки, errors, collisions |

### Диспетчер процессов CPU

Процесс в каждый момент находится в одном из трёх состояний:
1. **RUNNABLE** — готов к выполнению, ждёт своей очереди
2. **ON CPU** (USER/SYS) — выполняется
3. **WAIT FOR SMTHG** — ждёт I/O, блокировки, и т.п.

**Context switch (CS)** — переключение процессора с одного процесса на другой:
- **Voluntary CS** — процесс сам отдал управление (выполнил I/O)
- **Involuntary CS (preemption)** — ОС принудительно переключила (появился более приоритетный процесс)

## 7. Виртуальная память

**Виртуальная память** — метод управления памятью, позволяющий запускать программы, требующие больше RAM, чем есть физически. ОС автоматически перемещает страницы между RAM и диском (swap).

**Ключевые параметры:**
- **Scan rate** — частота сканирования страниц (высокий scan rate = мало RAM)
- **Page fault** — обращение к странице, которой нет в RAM:
  - *Minor* — нужно только создать запись в таблице адресации
  - *Major* — нужно загрузить страницу с диска (медленно!)
- **pgpgin/s, pgpgout/s** — страниц в/из RAM в секунду

## 8. Мониторинг Linux vs Windows

### Linux
Много специализированных инструментов:
- `top` / `htop` — онлайн-мониторинг процессов
- `vmstat` — виртуальная память, CPU, I/O
- `iostat` — статистика дисковой подсистемы
- `netstat` — сетевые соединения
- `sar` — исторические данные о системе
- `perf` — счётчики производительности процессора
- `strace` — трассировка системных вызовов (интрузивен)
- `lttng`, `stap (SystemTap)` — точки сбора в ядре (слабоинтрузивны)

### Windows
- **Task Manager** — базовый мониторинг
- **Resource Monitor & Performance Monitor** — более детально
- **Reliability Monitor** — история сбоев
- **Microsoft SysInternals** — самые точные данные (внутрисистемные счётчики)

## 9. Пример анализа vmstat

**Формат:** `vmstat 1 5` — 5 измерений с интервалом 1 секунда

```
Столбцы:
r  - процессы в очереди на выполнение
b  - процессы, заблокированные на I/O
us - % CPU на пользовательские программы
sy - % CPU на ядро ОС
id - % простоя CPU
wa - % CPU ждёт I/O
si/so - страницы из/в swap
bi/bo - блоки с/на диск
```

**Диагностика по vmstat:**
- `us = 100%` → CPU перегружен приложениями (профилировать!)
- `sy = 100%` → перегрузка ядра (много syscalls, дефект модуля ядра)
- `wa высокий + bo/bi большие + мало free` → нехватка RAM, активный своппинг

## 10. "Системный анализ за 60 секунд" (последовательность команд)

```bash
uptime              # load average за 1, 5, 15 минут
dmesg | tail        # последние ошибки ядра
vmstat 1            # память, swap, CPU
mpstat -P ALL 1     # загрузка по каждому CPU
pidstat 1           # горячие процессы
iostat -xz 1        # параметры дискового I/O
free -m             # свободная память/буферы/кэши
sar -n DEV 1        # сетевая статистика по интерфейсам
sar -n TCP,ETCP 1   # TCP статистика
top                 # общий онлайн-мониторинг
```

## 11. Мониторинг JVM: JIT + GC + Classloader

### Структура памяти JVM (heap):

```
┌─────────────────────────────────────────────────────────┐
│                    HEAP                                  │
│  ┌─────────────────────────┐  ┌───────────────────────┐│
│  │    Young Generation     │  │   Old Generation      ││
│  │  ┌───┬───┬─────────┐   │  │   (Tenured)           ││
│  │  │S0 │S1 │  Eden   │   │  │                       ││
│  │  └───┴───┴─────────┘   │  │                       ││
│  └─────────────────────────┘  └───────────────────────┘│
└─────────────────────────────────────────────────────────┘
     (в современных JDK Permanent Generation отсутствует)
```

**Как работает GC (сборщик мусора):**
1. Новые объекты → **Eden**
2. Прожившие Minor GC → **S0 или S1** (выжившие)
3. Долгожители → **Tenured (Old Generation)**
4. Число выживших GC до перехода в Old: от 8 до 32 (динамически)

**Параметры JVM:**
- `-Xms` / `-Xmx` — начальный/максимальный heap
- `-XX:NewSize` / `-XX:MaxNewSize` — Young Generation
- `-XX:+PrintGCDetails -verbose:gc` — подробный лог GC

**Инструменты:**
- `jps` — список Java-процессов
- `jstat -gcutil <pid> 1000` — статистика GC каждую секунду
- `JVisualVM + VerboseGC` — визуализация GC
- `JConsole`, `jmap`, `jhat`

## 12. Профилировщики приложений

**Профилировщик нужен для анализа:**
- Времени исполнения методов (что "горячее")
- Создания объектов в памяти (где утечки)
- Потоков и блокировок (contentions)

**Два подхода к профилированию:**

| Подход | Суть | Интрузивность |
|---|---|---|
| **Инструментирование** | Вставка кода в начало/конец методов | Высокая (только для подозрительных мест!) |
| **Сэмплирование** | Периодические прерывания + снимок стека | Низкая (не замедляет сильно) |

### NetBeans Profiler (пример):
- **Methods view** — методы, отсортированные по CPU времени. Можно найти `java.util.ArrayList.contains()` с 17% CPU.
- **Objects view** — объекты по занятой памяти. Растущий `byte[]` → утечка памяти (объекты не удаляются, GC не может освободить)

## 13. Trade-offs: алгоритмы и архитектура

**Золотое правило:** Быстрее доступ → больше памяти.

| Быстро | vs | Экономно |
|---|---|---|
| Индексирование | vs | Линейный поиск |
| Буферизация в памяти | vs | Прямое чтение с диска |
| Кэширование | vs | Без кэша |

> 💡 **Грамотный выбор алгоритма и архитектуры может ускорить приложение в миллионы раз!**

## 14. Рецепты: что делать при конкретных симптомах

### Высокий %SYS (ядро ОС перегружено)
- Высокий I/O → реже читать/писать, сжимать данные, увеличить блок передачи
- Много context switch → проверить количество потоков
- Paging/swapping → добавить RAM или запретить выгрузку критичных процессов

### Высокий %IO wait (процессор ждёт диска)
- Оптимизировать запросы к диску
- Расширить буферы/кэши
- Купить SSD (особенно при случайном доступе)

### Высокий %Idle (процессор простаивает)
- Мало RUNNABLE-потоков → **распараллелить** алгоритмы
- Добавить потоки в пулы
- Анализировать и оптимизировать блокировки (lock contention)
- Использовать lock-free алгоритмы

### Высокий %user (приложение жрёт CPU)
- Использовать алгоритмы с меньшей сложностью
- Повторно использовать объекты (не создавать/удалять крупные объекты)
- Убрать активные опросы в циклах (busy waiting)
- Минимизировать cache miss (группировать данные, паддинг)

## 15. Счётчики производительности CPU (perf)

В процессоре есть аппаратные регистры — **счётчики производительности**. Считают события:
- `cache-misses` — промахи кэша
- `L1-dcache-load-misses` — промахи L1 кэша данных
- `branch-misses` — ошибки предсказания переходов
- `cpu-cycles`, `instructions` — циклы и инструкции
- `dTLB-load-misses` — промахи TLB (буфера трансляции адресов)

**Использование в Linux:**
```bash
perf list | grep -i hardware    # список доступных событий
perf stat java MyApp            # статистика при запуске
perf record java MyApp          # запись профиля
perf report                     # анализ
```

```

Время обслуживания     — от запроса до ПОЛНОГО результата
Пропускная способность — макс. запросов В ЕДИНИЦУ ВРЕМЕНИ
Утилизация %util       — доля времени ресурс ЗАНЯТ
Точка насыщения        — после неё производительность ПАДАЕТ

Cache miss 2% → производительность < 60% от идеала!
Правильный алгоритм → ускорение в МИЛЛИОНЫ раз
```
