1. Платформа Java EE. Спецификации и их реализации. | Java EE 平台、规范与实现
- Java EE (Jakarta EE) — это набор стандартов (интерфейсов) для создания корпоративных приложений. Она определяет правила, которым должны следовать разработчики и производители серверов.

- Спецификации: Это технические документы (например, JPA для баз данных, JAX-RS для REST). Они описывают «что» должна делать технология, но не «как».

- Реализации (Application Servers): Это конкретное ПО, которое реализует эти спецификации.

  - Full Profile: GlassFish, Payara, WildFly, IBM WebSphere.
  - Servlet Containers (частичная реализация): Tomcat, Jetty (реализуют только Servlet/JSP).
规范与实现： Java EE 是一套标准（接口）。规范（Specification）是文档，而实现（Implementation）是具体代码。应用服务器如 WildFly 实现了这些规范。

2. Принципы IoC, CDI и Location Transparency. Компоненты и контейнеры. | IoC, CDI, 位置透明性, 组件与容器
- IoC (Inversion of Control): Принцип, при котором управление программой передается внешнему фреймворку (контейнеру). Не объект создает зависимости, а контейнер внедряет их в объект.

- CDI (Contexts and Dependency Injection): Спецификация Java EE для управления контекстами (жизненным циклом) и внедрения зависимостей. Основная аннотация — @Inject.

- Location Transparency (Прозрачность местоположения): Возможность вызывать компоненты (например, EJB) одинаково, независимо от того, находятся ли они в той же JVM или на удаленном сервере.

- Компоненты и Контейнеры: Компонент — это единица бизнес-логики (Servlet, EJB). Контейнер — это среда выполнения, которая предоставляет компонентам системные сервисы (безопасность, транзакции, жизненный цикл).

IoC 与 CDI： IoC 是设计思想（控制反转），CDI 是 Java EE 的具体依赖注入规范。位置透明性指调用远程组件像调用本地组件一样简单。

3. Управление жизненным циклом компонентов. Дескрипторы развёртывания. | 生命周期管理与部署描述符
- Жизненный цикл: Контейнер управляет созданием и уничтожением объектов.

  - @PostConstruct: Метод вызывается после того, как все зависимости внедрены.

  - @PreDestroy: Метод вызывается перед тем, как компонент будет удален из памяти.

- Дескрипторы развертывания (Deployment Descriptors): Конфигурационные файлы в формате XML.

  - web.xml: Настройки веб-модуля (сервлеты, фильтры).

  - ejb-jar.xml: Настройки EJB-компонентов.

  - beans.xml: Активация CDI.

生命周期： 容器负责组件的初始化和销毁。部署描述符是用于配置应用行为的 XML 文件。

4. Java EE API. Виды компонентов. Профили платформы Java EE. | Java EE API, 组件类型与配置文件
- Java EE API: Ключевые API включают JPA (БД), JTA (Транзакции), JMS (Сообщения), JSF (UI).
- Виды компонентов:
    1. Web-компоненты: Servlets, JSP, JSF.
    2. Бизнес-компоненты: EJB (Session Beans, Message-Driven Beans).

- Профили платформы:
  - Full Platform: Включает все спецификации (для тяжелых Enterprise-систем).
  - Web Profile: Облегченная версия для веб-приложений (без удаленных вызовов EJB, JMS и т.д.).

平台 Profiles： Web Profile 适用于轻量级 Web 应用，而 Full Platform 包含所有功能（如异步消息和远程调用）。

5. Компоненты EJB. Stateless & Stateful Session Beans. EJB Lite и EJB Full. | EJB 组件: 无状态/有状态, Lite/Full
- EJB (Enterprise JavaBeans): Компоненты для реализации бизнес-логики с поддержкой транзакций и безопасности.

- Stateless Session Beans: Не хранят состояние клиента между вызовами. Эффективны, так как контейнер использует пул объектов.

- Stateful Session Beans: Хранят состояние конкретного клиента (например, корзина покупок).

- EJB Lite: Минимальный набор функций EJB, входящий в Web Profile.

- EJB Full: Включает дополнительные возможности (удаленный доступ, MDB).

EJB 类型： Stateless（无状态）响应快，不保存数据；Stateful（有状态）为每个客户保存会话。

6. Работа с электронной почтой в Java EE. JavaMail API. | JavaMail API
- JavaMail API: Позволяет приложению отправлять и получать электронную почту.

- Основные классы:

  - Session: Объект конфигурации (хост, порт, аутентификация).

  - Message / MimeMessage: Представление письма.

  - Transport: Отправка сообщения (обычно через SMTP).

  - Store: Получение сообщений (через POP3/IMAP).

JavaMail： 用于邮件收发的标准接口。

7. JMS. Реализация очередей сообщений. Способы доставки. Message-Driven Beans. | JMS, 消息队列与 MDB
- JMS (Java Message Service): API для асинхронного обмена сообщениями.

- Модели доставки:

  - Point-to-Point (PTP): Сообщение идет в Queue (очередь) и читается одним получателем.

  - Publish/Subscribe (Pub/Sub): Сообщение идет в Topic (тему) и доставляется всем активным подписчикам.

- Message-Driven Beans (MDB): Специальный тип EJB, который слушает очереди/темы JMS и автоматически обрабатывает входящие сообщения асинхронно.

消息传递： JMS 定义了队列（点对点）和主题（发布/订阅）。MDB 是异步监听消息的组件。

8. Понятие транзакции. Управление транзакциями в Java EE. JTA. | 事务与 JTA
- Транзакция: Группа операций, которая выполняется как единое целое (ACID).

- JTA (Java Transaction API): Стандарт для управления распределенными транзакциями.

- Типы управления:

  - CMT (Container-Managed Transactions): Контейнер сам открывает и закрывает транзакции (через аннотацию @Transactional).

  - BMT (Bean-Managed Transactions): Программист вручную управляет транзакцией через API UserTransaction.

事务管理： JTA 是分布式事务标准。CMT 由容器自动管理，BMT 由开发者手动控制。

9. Веб-сервисы. Технологии JAX-RS и JAX-WS. | Web 服务: REST 与 SOAP
- JAX-WS: Используется для создания SOAP-сервисов. Базируется на XML и WSDL (сложные, строго типизированные сервисы).

- JAX-RS: Используется для создания RESTful-сервисов. Базируется на HTTP-методах (GET, POST, PUT, DELETE) и легких форматах (JSON, XML). Основная аннотация: @Path.

Web 服务： JAX-WS 适用于重量级的 SOAP 协议，JAX-RS 适用于现代的轻量级 REST API。

10. Платформа Spring. Сходства и отличия с Java EE. | Spring 与 Java EE 对比
- Сходства: Оба решают задачи корпоративного уровня, поддерживают IoC, DI, транзакции, работу с БД и безопасность.

- Отличия:

  - Java EE — это стандарт, требующий сервера приложений.

  - Spring — это фреймворк, который может работать в простом контейнере сервлетов (Tomcat) или как автономное приложение (Spring Boot).

  - Spring часто считается более гибким и модульным.

对比： Java EE 是官方标准，Spring 是事实上的行业标准。Spring 更加轻量且无需复杂的服务容器。

11.  Модули Spring. Архитектура Spring Runtime. Spring Security и Spring Data. | Spring 模块、架构、Security 与 Data
- Spring Runtime: Архитектура состоит из Core Container (Beans, Core, Context), AOP (Aspects), Data Access (JDBC, ORM), Web (MVC).

- Spring Security: Модуль для обеспечения аутентификации (кто вы?) и авторизации (что вам можно?). Защищает методы и URL-адреса.

- Spring Data: Упрощает доступ к данным. Позволяет создавать репозитории (интерфейсы), для которых Spring сам генерирует реализацию (например, JpaRepository).

Spring 模块： Spring Security 负责安全，Spring Data 简化了数据库访问逻辑（DAO 层）。

12. Реализация IoC и CDI в Spring. Сходства и отличия с Java EE. | Spring 中的 IoC 与 CDI
- Spring IoC: Реализуется через ApplicationContext. Объекты называются Beans.

- Способы внедрения: @Autowired (аналог @Inject в Java EE), @Resource.

- Отличия: Spring предлагает более мощную систему конфигурации (Java Config, XML, аннотации) и поддержку условий (@Conditional), чего в классическом CDI меньше.

依赖注入对比： Spring 使用 @Autowired，Java EE 使用 @Inject。Spring 的配置方式更多元。

13. Реализация REST API в Java EE и Spring. | REST API 实现对比
- Java EE (JAX-RS): Использует аннотации @Path, @GET, @Produces. Пример: Jersey, RestEasy.

- Spring (Spring MVC / Boot): Использует аннотацию @RestController. Методы помечаются @GetMapping, @PostMapping. Включает встроенную поддержку Jackson для JSON.

REST 实现： Java EE 常用 @Path，Spring 使用 @RestController 和 @RequestMapping。

14. React JS. Архитектура и основные принципы. | React JS 架构与原则
- Архитектура: React — это библиотека для создания UI на основе компонентов.

- Принципы:

  - Virtual DOM: React создает копию реального DOM в памяти, сравнивает изменения и обновляет только то, что нужно.

  - Однонаправленный поток данных: Данные передаются от родителя к дочерним элементам.

  - Declarative UI: Мы описываем, как интерфейс должен выглядеть, а не как его менять шаг за шагом.

React 原则： 虚拟 DOM 提高性能，数据流是单向的（从上到下）。

15.  Компоненты React. State & props. "Умные" и "глупые" компоненты. | React 组件: State, Props, 智能与木偶组件
- Props: Входные данные, которые передаются компоненту извне (только для чтения).

- State: Внутренние данные компонента, которые могут меняться со временем (вызывают рендер).

- Умные компоненты (Smart/Container): Отвечают за логику, запросы к API и хранение состояния.

- Глупые компоненты (Dumb/Presentational): Только отображают данные, полученные через props, и не имеют своей логики.

组件区别： Props 是外部传入的只读属性；State 是组件内部的可变状态。

16. Разметка страниц в React-приложениях. JSX. | JSX 语法
- JSX (JavaScript XML): Синтаксическое расширение JS. Позволяет писать HTML-подобный код прямо в JavaScript.

- Особенности:

  - Внутри { } можно писать любой JS-выражение.

  - Вместо class используется className.

  - Все теги должны быть закрыты.

JSX： 允许在 JS 中写 HTML，使组件结构更直观。

17.  Навигация в React-приложениях. ReactRouter. | React 路由
- ReactRouter: Стандартная библиотека для управления переходами между страницами без перезагрузки браузера (SPA).

- Компоненты:

  - `<BrowserRouter>`: Обертка для работы с историей браузера.

  - `<Routes>` и `<Route>`: Определение путей (URL) и соответствующих им компонентов.

  - `<Link>`: Вместо `<a>` для навигации без перезагрузки.

路由： ReactRouter 实现了单页应用的页面切换。

18.  Управление состоянием интерфейса. Redux. | Redux 状态管理
- Redux: Глобальное хранилище состояния для всего приложения.

- Три кита Redux:
  1. Store: Единственный источник истины (хранит все данные).
  2. Actions: Объекты, описывающие «что произошло» (например, INCREMENT).
  3. Reducers: Чистые функции, которые принимают старое состояние + action и возвращают новое состояние.

Redux 核心： Store（存储）、Action（动作）和 Reducer（根据动作更新状态的函数）。

19. Angular: архитектура и основные принципы. | Angular 架构与原则
- Архитектура: Angular — это полноценная платформа (framework), построенная на TypeScript. Использует модульную структуру.

- Принципы:

  - Two-way Data Binding: Автоматическая синхронизация данных между моделью и представлением.

  - Dependency Injection: Встроенная система внедрения зависимостей.

  - Directives: Расширение возможностей HTML (например, *ngIf, *ngFor).

Angular 架构： 它是基于 TypeScript 的全功能框架，支持双向数据绑定。

20.  Angular: модули, компоненты, сервисы и DI. | Angular: 模块、组件、服务与注入
- NgModule: Класс с декоратором @NgModule, который объединяет компоненты, директивы и сервисы в логические блоки.

- Component: Объединяет HTML-шаблон, стили и логику на TS.

- Service: Класс для общей бизнес-логики (например, работа с HTTP).

- DI (Dependency Injection): Angular внедряет сервисы в конструкторы компонентов автоматически.

Angular 组件： Module 组织应用，Component 控制视图，Service 处理逻辑并被注入 (DI)。

21. Angular: шаблоны страниц, жизненный цикл компонентов, подключение CSS. | Angular: 模板、生命周期与 CSS
- Templates: HTML с использованием синтаксиса Angular (интерполяция {{ }}, привязка свойств [ ]).

- Lifecycle Hooks:
  - ngOnInit: Вызывается при создании.
  - ngOnChanges: При изменении входных свойств.
  - ngOnDestroy: Перед удалением.

- CSS: Стили могут быть глобальными или инкапсулированными (только для одного компонента) через поле styleUrls в декораторе.

周期钩子： 最常用的是 ngOnInit（初始化）。

22.  Angular: клиент-серверное взаимодействие, формы и валидация. | Angular: HTTP, 表单与验证
- HttpClient: Модуль для выполнения AJAX-запросов. Основан на Observables (библиотека RxJS).

- Формы:
  - Template-driven: Логика в HTML-шаблоне (проще).
  - Reactive Forms: Логика в коде TS (более гибкие и тестируемые).

- Валидация: Встроенные валидаторы (Required, Email, MinLength) позволяют проверять корректность ввода данных пользователем.

数据交互： HttpClient 基于响应式编程 (RxJS)。表单验证分为模板驱动和响应式两种。
